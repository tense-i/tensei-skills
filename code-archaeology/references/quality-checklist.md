# 代码考古演进分析质量检查清单

> 用于输出前自检，确保演进分析文档的质量达标。
> 每项必须通过，未通过的必须补充完善后再输出。

---

## 一、结构完整性检查

### 1.1 必备章节

- [ ] **0. 仓库画像**：基本信息、历史统计、当前技术栈、开发节奏
- [ ] **1. 演进时间线**：分期总览、分期表、核心矛盾链
- [ ] **2. 逐阶段深度分析**：每个 Epoch 的架构快照 + 关键 commit + 方案引入
- [ ] **3. 方案迭代追踪**：至少 3 个核心方案的完整演化路径
- [ ] **4. 关键决策清单**：决策总览 + 重点决策深度分析
- [ ] **5. 架构演化对比**：v0 vs 当前态 + 至少 3 张架构快照
- [ ] **6. 技术债务时间线**：债务清单 + 模式分析
- [ ] **7. PR 全景分析**（如有数据）：分类统计 + 高价值 PR 分析
- [ ] **8. 演进叙事**：主线故事 + 转折点 + 踩坑记录
- [ ] **9. 总评与展望**：质量评估 + 模式总结 + 未来方向
- [ ] **附录**：关键提交索引 + 决策清单 + Mermaid 图索引 + 可复现命令

### 1.2 数量标准

| 检查项 | 最低标准 | 推荐标准 |
|--------|---------|---------|
| 篇幅 | 内容翔实完整 | 深入且全面 |
| Epoch 数 | ≥ 3（小项目）/ ≥ 5（中大型） | 5-12 |
| 关键提交卡片 | ≥ 5（小项目）/ ≥ 10 | 10-30 |
| 决策卡片 | ≥ 5 | ≥ 8 |
| 方案迭代追踪 | ≥ 3 个方案 | 3-5 |
| Mermaid 图 | ≥ 6 张 | ≥ 10 张 |

## 二、内容质量检查

### 2.1 视角正确性

- [ ] 是**方案演进分析**，不是 git log 翻译？
- [ ] 每段都在回答"方案从什么变到什么，为什么"？
- [ ] 是架构师复盘视角，不是开发者 debug 视角？
- [ ] 关注的是"方案意义"而不是"代码细节"？

### 2.2 证据标注

- [ ] 每条关键判断都标注了 `[证据]` `[推断]` `[假设]`？
- [ ] `[证据]` 是否附带 commit hash / 文件路径 / diff 片段？
- [ ] `[推断]` 是否说明了推断依据？
- [ ] `[假设]` 是否给出了验证建议？
- [ ] 是否**没有**把推断当事实写？

### 2.3 因果逻辑

- [ ] 每个 Epoch 是否说明了"核心矛盾"（为什么进入这个阶段）？
- [ ] Epoch 之间是否有因果连接（上一阶段遗留什么 → 推动进入下一阶段）？
- [ ] 演进叙事是否串成了有转折点的故事？
- [ ] 方案迭代追踪是否解释了"每次变化的驱动力"？

### 2.4 信息密度

- [ ] 每段是否都有实质性分析（非套话/空话）？
- [ ] 考古卡片的 "Why" 和 "Trade-off" 是否具体？
- [ ] 决策卡片的 "权衡" 是否说明了获得了什么、付出了什么？
- [ ] 是否**没有**用空行/重复内容凑篇幅？

---

## 三、Mermaid 图检查

### 3.1 必备图表

| 图类型 | 必须数量 | 用途 |
|--------|---------|------|
| 演进时间线 | ≥ 1 | timeline/gantt，展示阶段划分 |
| 架构快照 | ≥ 3 | graph，展示不同阶段的架构状态 |
| 方案演化图 | ≥ 2 | graph LR，展示方案从 V1→V2→V3 |
| 其他（可选） | - | 依赖变化图/技术栈变迁图 |

### 3.2 图表质量

- [ ] 每张图是否有标题说明？
- [ ] 每张图是否配合文字解读（图是"what"，文字是"why"）？
- [ ] Mermaid 语法是否正确（代码块围栏 ````mermaid`）？
- [ ] 图表是否在正确的章节位置？

---

## 四、演进分析特有检查

### 4.1 分期质量

- [ ] Epoch 是按"方案主题断层"切分的，不是按月份机械切分？
- [ ] 每个 Epoch 能否用一句话概括主题？
- [ ] 分期的起止 commit 是否明确？
- [ ] 是否遗漏了重要的演进阶段？

### 4.2 关键提交挑选

- [ ] 关键提交是否有"方案意义"（不是日常小修改）？
- [ ] 是否覆盖了多种类型（新方案/重构/迁移/修复/工程化）？
- [ ] 是否遗漏了明显的里程碑 commit？
- [ ] 考古卡片的 5 个字段是否都填充了？

### 4.3 方案迭代追踪

- [ ] 是否选择了系统的**核心方案**来追踪？
- [ ] 追踪是否覆盖了方案的完整生命周期（引入→迭代→稳定）？
- [ ] 每次演化是否解释了驱动力和取舍？

### 4.4 决策考古

- [ ] 决策是否有时间锚点（commit hash + 日期）？
- [ ] 是否分析了"被放弃的选项"？
- [ ] 是否分析了"决策的后续影响"？

### 4.5 技术债务

- [ ] 是否从 revert/hotfix/TODO/hack 等信号中识别了技术债务？
- [ ] 债务的引入阶段和清偿阶段是否标注？
- [ ] 是否分析了项目的债务管理模式？

---

## 五、避免事项检查

- [ ] 是否**没有**使用 `git checkout` / `git switch`？
- [ ] 是否**没有**逐条翻译 commit message？
- [ ] 是否**没有**只列文件变更不做方案解读？
- [ ] 是否**没有**深入代码实现细节（行级分析）？
- [ ] 是否**没有**只复述 README 不分析演进？
- [ ] 内容是否连贯完整，无断裂？
- [ ] 是否**没有**遗漏关键的架构性变更？
- [ ] 是否**没有**把 diff 大段粘贴代替分析？

---

## 六、输出完整性检查

### 6.1 多片衔接

- [ ] 跨片内容是否衔接自然？
- [ ] 每片末尾是否注明下一片覆盖内容？
- [ ] 所有章节是否在全部片段中完整覆盖？

### 6.3 附录完整性

- [ ] 关键提交索引是否与正文一致？
- [ ] 决策清单是否与正文一致？
- [ ] Mermaid 图索引是否覆盖所有图？
- [ ] 可复现命令是否可以直接运行？
- [ ] 技术栈变迁矩阵是否反映了完整演进？

---

## 七、评分维度参考

| 维度 | 评分标准 |
|------|---------|
| **演进洞察深度** | 是否揭示了表面变更背后的深层驱动力？ |
| **证据充分性** | 关键判断是否都有证据支撑？ |
| **因果叙事性** | 演进故事是否有清晰的因果逻辑和转折点？ |
| **方案追踪完整性** | 核心方案的演化路径是否完整？ |
| **决策分析深度** | 决策的权衡分析是否深入？ |
| **可复现性** | 读者能否用附录命令验证结论？ |
| **视觉呈现** | Mermaid 图+表格是否有效辅助理解？ |
