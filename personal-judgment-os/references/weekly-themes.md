# Weekly Theme Library

## A. High-Concurrency Transaction Systems

Use when the user wants market-visible backend/system design training.

Suggested 7-day map:

| Day | Scenario | Main judgment |
|---|---|---|
| Mon | Flash sale | Fairness and inventory consistency matter more than raw QPS. |
| Tue | Inventory | Reservation, deduction, rollback, and auditability define trust. |
| Wed | Order system | The state machine and idempotency are the core. |
| Thu | Payment | Accounting consistency and compensation are more important than UX smoothness. |
| Fri | Clearing/settlement | Auditability and reconciliation define correctness. |
| Sat | Risk control | The core trade-off is false positive vs false negative. |
| Sun | Integration | Degradation and compensation connect all modules. |

Expected asset: `High-Concurrency Transaction Systems Judgment Framework`.

## C. IM / Real-Time Systems

Suggested 7-day map:

| Day | Scenario | Main judgment |
|---|---|---|
| Mon | One-to-one chat | Message identity, ordering, and delivery semantics. |
| Tue | Group chat | Write amplification vs read amplification. |
| Wed | Presence | Freshness, cost, and correctness cannot all be maximized. |
| Thu | Offline messages | Sequence, ack, and replay define reliability. |
| Fri | Push | Reachability and annoyance are in tension. |
| Sat | Reliability | Exactly-once is usually a product illusion built on idempotency. |
| Sun | Integration | Real-time architecture is an end-to-end semantics problem. |

Expected asset: `Real-Time Communication Systems Judgment Framework`.
