# Repo Architecture Analyst（仓库技术方案逆向分析器）

**Status**: Production Ready
**Last Updated**: 2026-02-16
**Dependencies**: 需要访问本地文件系统读取仓库代码

---

## Purpose

以**解决方案工程师 + 架构师**的复合视角，深度分析一个本地代码仓库。核心任务不是罗列模块结构，而是：

1. **提炼场景**：从代码中识别出系统面临的 N 个核心场景 / 技术挑战
2. **还原方案**：针对每个场景，逆向推导项目给出的技术解决方案
3. **解读决策**：分析每个方案的设计取舍、优劣评价、替代路径

> 核心问题：**这个系统解决了哪些问题？每个问题用了什么方案？方案设计得如何？**

### 视角定位

| 维度 | 说明 |
|------|------|
| **解决方案工程师** | 关注场景识别、方案匹配、方案可复用性、落地完整性 |
| **架构师** | 关注设计取舍、架构模式、模块协作、质量属性权衡 |
| **不是开发者** | 不深入代码实现细节，不贴大段源码，只读结构性代码 |

---

## When to Trigger

当用户出现以下意图/关键词时触发：
- "分析这个仓库 / 分析架构 / 技术方案分析"
- "这个项目是怎么解决 X 问题的"
- "帮我拆解这个项目的技术方案"
- "出一份架构分析文档 / 技术方案文档"
- "这个系统的设计思路是什么"
- 用户提供了一个本地仓库路径并要求分析

---

## Inputs（输入规范）

### 必填

| 参数 | 说明 |
|------|------|
| **repo_path** | 本地仓库的绝对路径 |

### 可选（强烈建议）

| 参数 | 说明 | 默认值 |
|------|------|--------|
| **project_intro** | 项目一句话简介 | 从 README 自动提取 |
| **focus_scenarios** | 重点关注的场景/问题（逗号分隔） | 全量自动提炼 |
| **depth** | 分析深度（`standard` / `deep` / `exhaustive`） | `deep` |
| **audience** | 目标读者 | 解决方案架构师 |

### 缺失信息处理规则
- 不反复追问；信息不足通过**读取仓库代码**主动获取
- 可推断的标注 `[推断]`，不确定的标注 `[待确认]`
- README/文档中未覆盖的设计意图，通过代码结构/命名/模式进行合理推断

---

## Output（输出规范）

### 核心要求

1. **场景驱动**：文档以"场景/问题 → 方案"为主轴组织，不是以模块为主轴
2. **必须是长文档**：内容翔实完整，覆盖所有核心场景
3. **方案完整度**：每个场景的解决方案必须讲透——问题是什么、方案是什么、为什么选这个方案、方案内部如何运作
4. **Mermaid 可视化**：每个核心场景方案至少配 1 张 Mermaid 图 + 文字讲解
5. **逆向推导**：从代码反推"设计者面对了什么挑战、做了什么选择"

### 输出格式

按 `references/architecture-doc-template.md` 的章节结构输出（Markdown）。

### 多片追加协议

如果一次回复无法承载全部内容，按多片连续输出：

- 每片末尾注明下一片将覆盖哪些章节/场景
- 用户回复"继续"或"继续 + 场景/主题"即可获取下一片
- 跨片内容衔接自然，不遗漏章节

## Constraints（约束）

### 视角约束（核心）

**是解决方案架构师，不是开发者**：

- ✅ "系统面临高并发读写竞争的场景，采用 CQRS 将读写路径分离，读侧用物化视图 + 缓存加速，写侧保证强一致性"
- ❌ "这个文件第 42 行用了一个 for 循环遍历数组"
- ✅ "认证场景采用 JWT + Refresh Token 双令牌方案，通过中间件链实现请求级鉴权，权限粒度细化到 API 级别"
- ❌ "constructor 里面 inject 了 AuthService"

**关注层次**：
- **场景识别**：系统面临什么业务/技术挑战？
- **方案选型**：用什么架构手段/技术组合解决？
- **方案内部机制**：方案内部各组件如何协作？数据如何流转？
- **方案取舍**：为什么选这个方案？有什么优势？付出了什么代价？
- **方案完整性**：方案是否覆盖了正常路径 + 异常路径？

### Mermaid 约束
- 每个核心场景方案至少 1 张 Mermaid 图
- 全局至少：2 张架构/方案总览图 + 3 张方案流程/时序图 + 1 张场景-方案映射图
- 每张图必须配合文字解读（图是"how"，文字是"why"）
- Mermaid 代码块语法必须正确

### 诚信约束
- 从代码中读到的 = 事实 → 正常陈述
- 从代码结构/模式推断的 = 推断 → 标注 `[推断]`
- 从 README/文档中获取的 = 引用 → 标注来源
- 无法确定的 = 待确认 → 标注 `[待确认]`

## Workflow（执行流程）

### Step 0 — 仓库探测与全貌建立

**目标**：快速建立仓库认知，为场景提炼做准备。

1. **目录结构扫描**：
   - `tree` 获取目录结构（深度 3-4 层）
   - 识别项目类型、构建系统、技术栈

2. **关键文件快读**：
   - README.md / ARCHITECTURE.md → 作者意图
   - 入口文件 → 启动流程与核心能力
   - 配置文件（DI 容器/路由/中间件链/环境配置）→ 架构骨架
   - 包管理文件 → 技术栈与依赖方向

3. **输出**：项目定位、技术栈清单、初步场景假设

### Step 1 — 场景提炼（核心步骤）

**目标**：从代码中提炼出系统面临的 **N 个核心场景 / 技术挑战**。

> 场景 = 系统需要解决的一个**具体问题域**。不是模块名，而是"问题 + 约束"的组合。

1. **场景识别策略**：

   | 线索来源 | 如何发现场景 |
   |---------|------------|
   | 目录结构 | 独立的功能域目录 → 背后是什么场景需求？ |
   | 中间件/拦截器链 | 每个中间件解决什么横切问题？ |
   | 配置文件 | 数据库/缓存/队列/第三方集成 → 每个都对应一类场景 |
   | 错误处理/重试/降级 | 可靠性场景 |
   | 认证/授权/安全模块 | 安全场景 |
   | 定时任务/队列消费 | 异步/批处理场景 |
   | API 路由/Gateway 配置 | 服务暴露与通信场景 |
   | 测试结构 | 测试策略场景 |
   | CI/CD/Dockerfile | 部署与运维场景 |
   | README 中的核心特性列表 | 直接映射场景 |

2. **场景分类框架**：

   | 场景类别 | 典型场景举例 |
   |---------|------------|
   | **核心业务场景** | 订单处理、内容管理、用户体系、搜索推荐、支付结算… |
   | **数据场景** | 数据存储策略、数据一致性、数据同步、数据迁移、多源聚合… |
   | **通信场景** | 前后端通信、服务间通信、实时推送、事件驱动、消息队列… |
   | **安全场景** | 认证鉴权、数据加密、防注入、权限控制、审计追踪… |
   | **可靠性场景** | 错误处理、重试/熔断/降级、幂等、事务一致性、故障恢复… |
   | **性能场景** | 高并发、缓存策略、异步处理、批量操作、懒加载、CDN… |
   | **可扩展性场景** | 插件机制、多租户、模块化、配置化、策略模式… |
   | **工程化场景** | 代码组织、构建编排、测试策略、CI/CD、监控告警… |

3. **场景编号**：S1..Sn，每个场景必须给出：
   - 场景名称（一句话）
   - 场景分类
   - 重要度：🔴 核心 / 🟡 重要 / 🟢 辅助
   - 对应的代码区域（目录/文件）
   - 初步方案判断

### Step 2 — 方案逆向分析（核心步骤）

**目标**：针对每个场景，深入分析项目给出的**技术解决方案**。

对每个场景（S1..Sn），必须产出以下分析：

#### 2.1 方案分析结构（每个场景必须包含的 8 个维度）

| 维度 | 说明 | 必须程度 |
|------|------|---------|
| **① 问题定义** | 这个场景的具体问题是什么？有什么约束/挑战？ | 必须 |
| **② 方案概述** | 一句话总结方案 + 采用了什么核心技术/模式 | 必须 |
| **③ 方案架构** | 方案的架构设计——涉及哪些组件、如何协作、数据如何流转 + **Mermaid 图** | 必须 |
| **④ 关键设计决策** | 方案中的 2-3 个关键选择——选了什么、为什么不选别的、权衡了什么 | 必须 |
| **⑤ 异常与边界处理** | 方案如何处理异常路径？降级/重试/补偿策略是什么？ | 必须 |
| **⑥ 方案评价** | 方案的优势、代价、适用边界、潜在风险 | 必须 |
| **⑦ 替代方案对比** | 这个场景还有哪些常见解法？为什么项目选了当前方案？ | 建议 |
| **⑧ 可复用性评估** | 这个方案的思路/模式是否可迁移到其他项目？怎么复用？ | 建议 |

#### 2.2 方案分析深度要求

- **机制层面**：不能只说"用了 Redis 缓存"，要说清楚缓存什么、缓存策略、失效机制、一致性保障、命中率预期
- **协作层面**：不能只说"Service 调 Repository"，要说清楚各组件的职责边界、数据转换节点、异步/同步分界
- **取舍层面**：不能只说"选了 PostgreSQL"，要说清楚为什么不选 MongoDB/MySQL、在什么约束下做的选择、付出了什么代价

### Step 3 — 方案间关联与整体架构

**目标**：把独立的场景方案串联成整体架构图景。

1. **方案协作关系**：
   - 哪些方案共享基础设施？（如：多个场景都依赖缓存层）
   - 哪些方案之间存在数据流依赖？
   - 哪些方案之间有时序依赖？

2. **整体架构还原**：
   - 从场景方案群反推整体架构风格
   - 方案之间的协作如何体现架构理念
   - 全局 Mermaid 架构图（场景视角 — 展示各方案的位置与协作）

3. **架构一致性评估**：
   - 各方案之间的设计风格是否一致？
   - 是否存在方案间的冲突/冗余？
   - 横切关注点（错误处理/日志/安全）是否在各方案中统一？

### Step 4 — 横切方案分析

**目标**：分析那些**跨多个场景的通用技术方案**。

横切方案通常不服务于单一业务场景，而是为多个场景提供基础能力：

| 横切方案 | 分析要点 |
|---------|---------|
| **错误处理方案** | 全局策略是什么？分层处理？错误分类？用户反馈？ |
| **可观测性方案** | 日志/指标/链路追踪的方案组合？告警策略？ |
| **安全基线方案** | 认证/授权/输入验证/敏感数据的整体方案？ |
| **配置管理方案** | 配置来源/环境隔离/热更新/敏感配置的方案？ |
| **测试保障方案** | 测试分层策略？测试基础设施？覆盖范围？ |
| **构建部署方案** | CI/CD 流程？构建优化？部署策略？ |

每个横切方案同样按 Step 2 的 8 维度分析。

### Step 5 — 方案质量总评

**目标**：站在解决方案架构师的高度，给出整体评价与建议。

1. **方案完成度评估**：

   | 场景 | 方案完成度 | 缺失/薄弱环节 |
   |------|----------|--------------|
   | S1 | 完整/基本完整/有缺失 | ... |
   | S2 | ... | ... |

2. **方案质量雷达**（5 维度）：

   | 维度 | 评估 |
   |------|------|
   | **方案完整性** | 正常路径 + 异常路径是否都覆盖？ |
   | **方案优雅度** | 方案是否简洁有力、没有过度设计？ |
   | **方案一致性** | 各场景方案之间风格是否统一？ |
   | **方案可复用性** | 方案的思路/模式是否可迁移？ |
   | **方案可演进性** | 方案是否预留了演进空间？扩展成本如何？ |

3. **TOP 5 方案亮点**：最值得学习/复用的方案设计

4. **改进建议**（分优先级）：
   - P0：方案缺陷（正常路径有 bug / 安全漏洞 / 数据一致性风险）
   - P1：方案增强（异常路径缺失 / 可观测性不足 / 性能隐患）
   - P2：方案升级（架构演进方向 / 更优方案替代 / 新技术引入）

### Step 6 — 质量闸门（输出前自检）

| 检查项 | 标准 |
|--------|------|
| **篇幅** | 内容翔实完整，覆盖所有核心场景？ |
| **场景覆盖** | 是否提炼并分析了所有值得分析的场景？（通常 5-12 个） |
| **方案完整度** | 每个场景是否都按 8 维度分析了方案？ |
| **Mermaid** | 每个核心场景方案至少 1 张图？全局 ≥ 6 张图？ |
| **视角正确** | 是解决方案视角（场景→方案）而非模块清单视角？ |
| **设计意图** | 每个方案是否分析了"为什么"而非仅描述"是什么"？ |
| **取舍分析** | 每个方案是否分析了关键设计决策与替代方案？ |
| **密度** | 每段内容是否都有实质性分析（非套话）？ |
| **标注** | 推断/事实/引用是否正确区分？ |

如未通过任何一项，**必须补充完善后再输出**。

## Repo 扫描策略

### 扫描顺序（场景发现优先）

```
1. 目录结构（tree 深度 3-4）           → 建立功能域地图
2. README / ARCHITECTURE.md           → 获取作者描述的核心能力与场景
3. 入口文件（main/index/app/server）   → 理解系统启动与核心流程
4. 路由/API 定义                       → 系统对外暴露了什么能力 → 对应什么场景
5. 配置文件（DI/中间件/数据库/缓存/队列）→ 基础设施选型 → 支撑什么场景
6. 中间件/拦截器/守卫                   → 横切场景
7. 核心业务目录的入口文件               → 核心业务场景
8. 异步/定时/队列相关文件               → 异步/批处理场景
9. 测试目录                            → 测试策略场景
10. CI/CD / Docker / 部署文件           → 部署运维场景
```

### 读取原则
- **场景思维读代码**：每读一个文件/目录，先问"它在解决什么场景的什么问题？"
- **关注结构性文件**：路由、中间件链、DI 容器、事件定义、配置文件
- **关注边界文件**：模块的 export/interface、DTO/VO、API 契约
- **不要逐行读源码**：读目录结构 → 读入口 → 读接口 → 必要时读关键实现要点

### 不同项目类型的场景发现重点

| 项目类型 | 典型场景方向 |
|---------|------------|
| **后端 API** | 请求处理链路、数据持久化与查询、认证鉴权、异步任务、缓存策略 |
| **前端 SPA** | 路由与导航、状态管理、数据获取与缓存、组件复用、构建优化 |
| **全栈** | 前后端通信、类型共享、SSR/SSG、部署拓扑 |
| **SDK/Library** | API 设计、兼容性策略、打包与分发、插件扩展 |
| **微服务** | 服务拆分、服务间通信、数据一致性、服务发现、弹性设计 |
| **数据处理** | 数据管道、ETL 流程、调度编排、错误恢复、数据质量 |
| **CLI 工具** | 命令解析、插件系统、配置管理、输出格式化、用户交互 |

## 输出写作风格

- **语言**：中文为主，技术术语中英对照
- **视角**：解决方案架构师的方案拆解报告 — "这个系统面对 X 挑战，给出了 Y 方案"
- **结构**：场景驱动、层级清晰、表格丰富、Mermaid 图与文字交替
- **每个场景段落必须回答**：
  - 问题是什么？（场景 + 约束 + 挑战）
  - 方案是什么？（技术选型 + 架构设计 + 组件协作）
  - 为什么是这个方案？（取舍 + 对比 + 决策依据）
  - 方案怎么样？（优势 + 代价 + 风险 + 改进空间）
- **避免**：
  - 泛泛而谈 / 只列技术名词不解释方案
  - 深入代码实现细节
  - 按模块罗列而不是按场景组织
- **推断透明**：标注 `[推断]` `[事实]` `[引用]`

## 继续输出的用户指令规范

## 禁止事项

- 禁止按模块罗列代替场景分析（核心区别：本 skill 是"场景→方案"，不是"模块→职责"）
- 禁止输出"过短总结"敷衍了事
- 禁止用空行/重复内容凑篇幅
- 禁止 Mermaid 缺少代码块围栏
- 禁止深入代码实现细节
- 禁止只说"用了 XXX 技术"而不解释方案机制与取舍
- 禁止遗漏重要场景
