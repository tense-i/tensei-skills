# 架构模式与设计模式识别目录

> 在分析仓库时，按此目录逐一检测，识别代码中运用的架构模式和设计模式。
> 每种模式给出：识别信号（代码中的特征）→ 确认方法 → 分析要点。

---

## 一、架构风格（全局级）

### 1. 分层架构（Layered Architecture）

**识别信号**：
- 目录按 controller / service / repository / model 分层
- 明确的调用方向：上层 → 下层
- 禁止跨层调用（或通过 façade 跨层）

**确认方法**：
- 检查 import 方向是否单向
- 是否有"跨层直达"的坏味道

**分析要点**：
- 分几层？每层职责是否清晰？
- 层间是否有接口隔离？
- 是否严格/宽松分层？

---

### 2. 六角形架构 / 端口与适配器（Hexagonal / Ports & Adapters）

**识别信号**：
- 核心域（domain/core）不依赖外部框架
- 端口（port/interface）定义在核心层
- 适配器（adapter/infra）实现端口
- 依赖方向：外 → 内

**确认方法**：
- 核心域目录是否有 import 外部框架？
- 是否定义了 Port/Interface 被外层实现？

**分析要点**：
- 端口抽象是否合理？粒度如何？
- 核心域的可移植性如何？
- 适配器是否易于替换？

---

### 3. 洋葱架构 / 整洁架构（Clean Architecture）

**识别信号**：
- 依赖方向：外圈 → 内圈
- 内圈：实体/用例；外圈：框架/驱动/接口
- 用例层（use-case/application）独立于框架

**确认方法**：
- 检查各层 import 的依赖方向
- use-case 层是否可以脱离框架运行？

**分析要点**：
- 圈层是否分明？
- 依赖规则是否被严格执行？
- 是否有违反依赖规则的"漏洞"？

---

### 4. 微服务架构（Microservices）

**识别信号**：
- 独立可部署的多个服务目录
- 服务间通过 HTTP/gRPC/消息队列通信
- 每个服务有独立的数据存储
- Docker/K8s 配置

**确认方法**：
- 是否有多个 main 入口？
- 服务间是否通过网络通信？
- 数据存储是否独立？

**分析要点**：
- 服务边界是否合理？
- 服务间通信方式？
- 数据一致性策略？
- 服务发现与负载均衡？

---

### 5. 事件驱动架构（Event-Driven Architecture）

**识别信号**：
- 事件定义文件（events/）
- 事件总线/消息队列集成
- 发布-订阅模式
- 事件处理器/监听器

**确认方法**：
- 是否有 EventEmitter / EventBus / Message Queue 使用？
- 模块间是否通过事件解耦？

**分析要点**：
- 事件设计：命名/粒度/schema
- 事件流：谁发/谁收/什么链路
- 一致性：最终一致性 vs 强一致性
- 错误处理：死信队列/重试/补偿

---

### 6. CQRS（Command Query Responsibility Segregation）

**识别信号**：
- 读模型和写模型分离
- 不同的数据存储用于读和写
- Command/Query 分离的接口设计

**确认方法**：
- 是否有 Command/Query 分离的类/接口？
- 读写是否走不同的数据路径？

**分析要点**：
- 读写分离的粒度？
- 数据同步策略？
- 是否配合 Event Sourcing？

---

### 7. 插件架构（Plugin Architecture）

**识别信号**：
- 插件注册/发现机制
- 扩展点（Extension Point）定义
- 插件接口/协议
- 动态加载

**确认方法**：
- 是否有 plugin/extension 目录或注册表？
- 是否可以不修改核心代码添加功能？

**分析要点**：
- 插件 API 设计质量
- 插件隔离机制
- 生命周期管理

---

### 8. Monorepo 架构

**识别信号**：
- 工作空间配置（workspaces/lerna/turborepo/nx）
- packages/ 或 apps/ 多包结构
- 共享依赖和内部包引用

**确认方法**：
- 根目录是否有 workspace 管理工具配置？
- 包间是否有内部引用？

**分析要点**：
- 包划分策略
- 构建编排
- 版本策略
- 变更影响范围管理

---

## 二、设计模式（模块级）

### 创建型模式

| 模式 | 识别信号 | 常见位置 |
|------|---------|---------|
| **Factory** | `createXxx()` / `XxxFactory` / 集中创建逻辑 | 对象创建热点 |
| **Builder** | 链式调用 `.setA().setB().build()` | 复杂对象构建 |
| **Singleton** | 全局单例 / IoC 容器 Singleton 范围 | 资源管理器/连接池 |
| **Pool** | 对象池 / 连接池 / 线程池 | 高性能场景 |

### 结构型模式

| 模式 | 识别信号 | 常见位置 |
|------|---------|---------|
| **Adapter** | 将接口 A 转换为接口 B | 外部系统集成 |
| **Decorator** | 包装已有功能添加新行为 / `@decorator` | 中间件/增强 |
| **Facade** | 简化复杂子系统的统一接口 | 模块入口 |
| **Proxy** | 代理访问 / 延迟加载 / 缓存代理 | 远程调用/缓存 |
| **Composite** | 树形结构 / 统一叶子和组合操作 | UI 组件/规则引擎 |

### 行为型模式

| 模式 | 识别信号 | 常见位置 |
|------|---------|---------|
| **Strategy** | 可替换的算法/策略接口 | 业务规则/算法选择 |
| **Observer** | 事件监听 / 发布订阅 / 回调注册 | 事件系统/状态变更 |
| **Command** | 命令对象封装操作 / undo-redo | 操作队列/事务 |
| **Chain of Responsibility** | 中间件链 / 过滤器链 / 管道 | 请求处理/验证 |
| **State Machine** | 状态枚举 / 状态转换表 / 守卫条件 | 工作流/订单状态 |
| **Template Method** | 基类定义骨架 / 子类重写步骤 | 框架扩展点 |
| **Visitor** | 双分派 / 操作与对象结构分离 | 编译器/AST 处理 |

### 架构型模式

| 模式 | 识别信号 | 常见位置 |
|------|---------|---------|
| **Repository** | 数据访问抽象 / 集合语义 | 数据层 |
| **Unit of Work** | 事务跟踪 / 批量提交 | 数据层 |
| **Specification** | 可组合的查询条件/业务规则 | 查询/验证 |
| **Domain Event** | 领域事件定义与分发 | 领域层 |

### 并发模式

| 模式 | 识别信号 | 常见位置 |
|------|---------|---------|
| **Actor** | 消息传递 / 状态隔离 / 信箱 | 分布式/并发 |
| **Future/Promise** | 异步结果封装 | 全局 |
| **Producer-Consumer** | 队列 / 工作线程 / 任务分发 | 后台处理 |
| **Circuit Breaker** | 熔断器 / 状态切换(关/开/半开) | 外部调用 |

## 三、分析输出标准

对于每个识别到的模式，必须回答：

1. **在哪里？**：具体的目录/文件/模块
2. **解决什么问题？**：该模式在此项目中解决的具体问题
3. **实现质量？**：实现是否规范/有无变体/有无坏味道
4. **是否恰当？**：是否是最佳选择？有无过度设计？
5. **与其他模式的协作？**：是否与其他模式配合使用

---

## 四、模式识别优先级

在有限时间内优先识别：

1. **架构风格**（最重要 — 决定了整体设计方向）
2. **核心业务流程中的模式**（策略/状态机/命令/事件）
3. **数据层模式**（Repository/UoW/CQRS）
4. **基础设施模式**（中间件链/熔断器/缓存代理）
5. **创建型和结构型模式**（补充完善）
