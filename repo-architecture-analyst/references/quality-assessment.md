# 方案质量评估框架

> 用于评估代码仓库中各场景技术方案的质量。
> 视角：解决方案架构师，关注方案的完整性、优雅度、可复用性和可演进性。

---

## 一、方案质量五维度

### 1. 方案完整性（Completeness）

> 方案是否完整地解决了场景的全部问题？

| 等级 | 评分 | 标准 |
|------|------|------|
| 优秀 | ★★★★★ | 正常路径 + 全部异常路径 + 边界条件 + 降级策略 + 监控告警全覆盖 |
| 良好 | ★★★★☆ | 正常路径完备，主要异常路径覆盖，有基本降级策略 |
| 中等 | ★★★☆☆ | 正常路径完备，部分异常路径有处理 |
| 较差 | ★★☆☆☆ | 正常路径基本可用，异常路径大量缺失 |
| 糟糕 | ★☆☆☆☆ | 正常路径有不完整的地方，异常路径几乎没有 |

**评估清单**：
- [ ] 正常路径是否完整？
- [ ] 异常路径覆盖了哪些？遗漏了哪些？
  - 网络超时/中断
  - 数据不一致/脏数据
  - 并发冲突
  - 资源耗尽（内存/连接/队列）
  - 外部依赖不可用
  - 权限/认证失败
- [ ] 是否有降级/兜底策略？
- [ ] 是否有监控/告警？

---

### 2. 方案优雅度（Elegance）

> 方案是否简洁有力、恰到好处？

| 等级 | 评分 | 标准 |
|------|------|------|
| 优秀 | ★★★★★ | 方案简洁优美，恰到好处地使用模式，没有多余的复杂度 |
| 良好 | ★★★★☆ | 方案清晰合理，偶有代码上稍嫌繁琐但不影响理解 |
| 中等 | ★★★☆☆ | 方案能工作，但存在一些不必要的复杂度或过度设计 |
| 较差 | ★★☆☆☆ | 方案过度设计 或 过于简陋，不够平衡 |
| 糟糕 | ★☆☆☆☆ | 方案混乱，难以理解和维护 |

**评估要点**：
- 是否存在过度设计？（引入了问题规模不需要的模式/抽象）
- 是否存在设计不足？（应该但没有抽象的地方）
- 组件职责是否清晰？
- 数据流是否直观可追踪？
- 是否利用了框架/语言的惯用法？

---

### 3. 方案一致性（Consistency）

> 各场景方案之间的设计风格是否统一？

| 等级 | 评分 | 标准 |
|------|------|------|
| 优秀 | ★★★★★ | 所有方案遵循统一的架构约定，风格高度一致 |
| 良好 | ★★★★☆ | 大部分方案风格一致，个别场景有合理的差异 |
| 中等 | ★★★☆☆ | 核心方案风格一致，辅助方案风格有偏差 |
| 较差 | ★★☆☆☆ | 方案间风格差异明显，看起来像不同人/不同时期写的 |
| 糟糕 | ★☆☆☆☆ | 完全没有统一的架构约定 |

**评估要点**：
- 错误处理策略是否跨方案统一？
- 数据访问模式是否一致？
- 日志/监控方式是否统一？
- 命名/目录约定是否一致？
- 技术选型是否存在重复/冲突？（如同时用了两套 HTTP 客户端）

---

### 4. 方案可复用性（Reusability）

> 方案的思路/模式是否可迁移到其他项目？

| 等级 | 评分 | 标准 |
|------|------|------|
| 优秀 | ★★★★★ | 方案设计通用，可直接作为模板应用到同类项目 |
| 良好 | ★★★★☆ | 方案核心思路可复用，需少量适配 |
| 中等 | ★★★☆☆ | 部分设计可复用，但与项目特定逻辑耦合较紧 |
| 较差 | ★★☆☆☆ | 与项目强耦合，复用成本高 |
| 糟糕 | ★☆☆☆☆ | 完全为特定项目定制，无法复用 |

**评估要点**：
- 方案中的架构模式是否通用？
- 是否有清晰的接口/抽象层可以替换实现？
- 与业务逻辑的耦合度如何？
- 配置化/参数化程度如何？

---

### 5. 方案可演进性（Evolvability）

> 方案是否预留了演进空间？扩展成本如何？

| 等级 | 评分 | 标准 |
|------|------|------|
| 优秀 | ★★★★★ | 预留了扩展点，新增场景/需求只需添加而不修改 |
| 良好 | ★★★★☆ | 主要扩展路径已预留，新增需求影响范围小 |
| 中等 | ★★★☆☆ | 部分可扩展，部分需要重构才能支持新需求 |
| 较差 | ★★☆☆☆ | 大部分变更需要修改核心逻辑 |
| 糟糕 | ★☆☆☆☆ | 牵一发动全身，几乎无法安全扩展 |

**评估要点**：
- 是否遵循开闭原则？
- 扩展点设计是否足够？
- 依赖方向是否支持独立演进？
- 数据模型是否支持向前兼容？
- 是否有版本化策略？

## 二、方案改进建议分级

| 级别 | 含义 | 典型内容 |
|------|------|---------|
| **P0 方案缺陷** | 正常功能有bug / 安全漏洞 / 数据丢失风险 | 必须立即修复 |
| **P1 方案增强** | 异常路径缺失 / 可观测性不足 / 性能瓶颈 | 应在近期迭代解决 |
| **P2 方案升级** | 架构演进方向 / 更优方案替代 / 技术债清理 | 纳入技术路线图 |

### 建议格式

每条建议必须包含：

```
[场景编号] [优先级]
问题：当前方案在什么方面有什么问题？
影响：如果不改会怎样？
建议：应该怎么改？
路径：改动涉及哪些模块？预估工作量？
收益：改完之后有什么好处？
```

## 三、方案评估中的常见误区

| 误区 | 正确做法 |
|------|---------|
| 以代码量评判方案质量 | 以问题解决效果评判 |
| 认为"用了高级模式"就是好方案 | 评估模式是否匹配问题规模 |
| 只看正常路径 | 异常路径的处理才体现方案成熟度 |
| 只评价技术实现 | 还要评价方案对业务需求的匹配度 |
| 忽略运维视角 | 一个部署困难、不可观测的方案不是好方案 |
| 以最新技术为最好 | 稳定、团队熟悉的技术可能是最佳选择 |

## 四、评估输出检查清单

- [ ] 5 个维度是否都有评分和具体依据？
- [ ] 评分是否基于代码事实而非主观臆测？
- [ ] 每个场景方案是否都有独立评价？
- [ ] 改进建议是否按优先级分级？
- [ ] 每条建议是否包含"问题→影响→建议→路径→收益"？
- [ ] TOP 5 亮点是否给出了可学习的原因？
- [ ] 评估是否客观——优缺点是否对等呈现？
- [ ] 是否考虑了项目的约束条件（团队规模/技术栈/时间压力）再做评价？
