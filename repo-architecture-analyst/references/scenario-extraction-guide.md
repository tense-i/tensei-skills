# 场景提炼指南

> 本指南用于指导如何从代码仓库中系统性地提炼出技术场景。
> 场景 = 系统需要解决的一个具体问题域（问题 + 约束的组合），不等于模块名。

---

## 一、什么是"场景"

### 场景 vs 模块

| 概念 | 定义 | 举例 |
|------|------|------|
| **模块** | 代码的组织单元 | `auth/`、`user/`、`order/` |
| **场景** | 系统需要解决的问题域 | "如何在无状态服务中实现安全的用户认证" |

**同一个模块可能涉及多个场景**：
- `auth/` 模块可能同时涉及"认证场景"和"权限控制场景"

**同一个场景可能跨多个模块**：
- "数据一致性场景"可能跨越 `order/`、`payment/`、`inventory/` 多个模块

### 好的场景描述 vs 坏的场景描述

| ❌ 坏的（太模糊/就是模块名） | ✅ 好的（问题 + 约束） |
|---------------------------|---------------------|
| "用户模块" | "如何实现多租户环境下的用户隔离与共享数据管理" |
| "缓存" | "如何在高并发读取场景下通过多级缓存策略降低数据库压力" |
| "消息队列" | "如何通过异步消息保证订单与库存之间的最终一致性" |
| "错误处理" | "如何实现分层错误处理，在不泄露内部信息的前提下给用户友好反馈" |

## 二、场景识别的 10 条线索

### 线索 1：目录结构中的功能域

```
src/
├── auth/        → 认证鉴权场景
├── order/       → 订单处理场景
├── payment/     → 支付结算场景
├── notification/ → 消息通知场景
├── search/      → 搜索场景
└── analytics/   → 数据分析场景
```

**提炼方法**：对每个功能域目录，问"它在解决什么问题？这个问题有什么约束？"

### 线索 2：中间件链 / 拦截器

```
中间件链: cors → rateLimit → auth → validate → log → handler
```

每个中间件背后都是一个横切场景：
- CORS → 跨域访问控制场景
- RateLimit → 流量控制/防滥用场景
- Auth → 认证鉴权场景
- Validate → 输入验证场景
- Log → 可观测性场景

### 线索 3：数据存储配置

```yaml
databases:
  primary: PostgreSQL    → 关系型数据存储场景
  cache: Redis          → 缓存策略场景
  search: Elasticsearch → 全文搜索场景
  queue: RabbitMQ       → 异步消息场景
  storage: S3           → 文件存储场景
```

每个存储选型背后都有一个数据场景。

### 线索 4：外部服务集成

```
integrations/
├── stripe/      → 支付集成场景
├── sendgrid/    → 邮件通知场景
├── twilio/      → SMS 场景
├── aws-s3/      → 文件存储场景
└── openai/      → AI 能力集成场景
```

每个外部集成 = 一个服务集成场景（API 设计/错误处理/降级策略）。

### 线索 5：异步/定时/队列

```
jobs/
├── cron/        → 定时任务调度场景
├── workers/     → 后台处理场景
└── consumers/   → 消息消费场景
```

### 线索 6：错误处理/重试/降级代码

- 全局异常过滤器 → 统一错误处理场景
- 重试装饰器/中间件 → 弹性设计场景
- 降级/fallback 逻辑 → 高可用场景
- 熔断器配置 → 外部依赖保护场景

### 线索 7：认证/安全相关

- JWT/Session/OAuth 配置 → 认证场景
- RBAC/ABAC 权限模型 → 授权场景
- 加密/脱敏逻辑 → 数据安全场景
- CSRF/XSS 防护 → 安全防护场景

### 线索 8：测试结构

```
tests/
├── unit/        → 单元测试策略
├── integration/ → 集成测试策略
├── e2e/         → 端到端测试策略
└── fixtures/    → 测试数据管理
```

测试结构反映了团队的质量保障场景。

### 线索 9：CI/CD 与部署配置

- Dockerfile/Compose → 容器化部署场景
- GitHub Actions/GitLab CI → 持续集成场景
- Kubernetes 配置 → 编排与伸缩场景
- 环境配置文件 → 多环境管理场景

### 线索 10：README 中的特性列表

README 中列出的核心特性/功能，几乎每条都对应一个场景。

## 三、场景分类框架

### 按业务/技术层次分类

| 层次 | 场景类别 | 说明 |
|------|---------|------|
| **L1 核心业务** | 核心业务场景 | 系统存在的根本理由——核心业务流程 |
| **L2 业务支撑** | 数据/通信/安全场景 | 支撑业务运转的技术基础设施 |
| **L3 质量保障** | 可靠性/性能/可扩展性 | 保证系统在生产环境稳定运行 |
| **L4 工程效能** | 测试/CI/CD/监控 | 保证团队高效开发和运维 |

### 按常见领域分类

| 场景类别 | 典型场景 |
|---------|---------|
| **核心业务** | 订单/支付/用户/内容/搜索/推荐/社交/工作流 |
| **数据管理** | 存储策略/查询优化/一致性/迁移/聚合/ETL |
| **服务通信** | API 设计/服务间调用/实时推送/事件驱动/消息 |
| **安全防护** | 认证/授权/加密/审计/合规/防攻击 |
| **可靠性** | 错误处理/重试/熔断/降级/幂等/事务 |
| **性能优化** | 缓存/异步/批量/CDN/懒加载/连接池 |
| **可扩展性** | 插件化/多租户/模块化/配置化/策略模式 |
| **工程化** | 代码组织/测试/CI/CD/监控/日志/配置管理 |

## 四、场景提炼输出格式

对每个提炼出的场景，输出：

```
编号：S{n}
名称：[一句话问题描述]
分类：[类别]
重要度：🔴 核心 / 🟡 重要 / 🟢 辅助
问题：[这个场景的具体问题 + 约束 + 挑战]
代码区域：[对应的目录/文件/模块]
初步方案判断：[从代码中初步观察到的解决手段]
```

## 五、场景提炼的常见误区

| 误区 | 正确做法 |
|------|---------|
| 把模块名当场景名 | 问"这个模块解决什么问题" |
| 粒度太粗（整个系统当一个场景） | 拆到可以独立分析方案的粒度 |
| 粒度太细（每个函数一个场景） | 合并到有意义的问题域级别 |
| 只关注核心业务场景 | 横切场景（安全/可靠性/性能）同样重要 |
| 只关注正常路径 | 异常/降级/恢复本身就是场景 |
| 遗漏工程化场景 | 测试/CI/CD/监控也是重要场景 |

## 六、场景数量参考

| 项目规模 | 建议场景数 |
|---------|----------|
| 小型项目（< 5000 行） | 3-5 个 |
| 中型项目（5000-50000 行） | 5-8 个 |
| 大型项目（50000-200000 行） | 8-12 个 |
| 超大型项目（> 200000 行） | 10-15+ 个 |

**原则**：宁可多提炼再合并，不要遗漏重要场景。
