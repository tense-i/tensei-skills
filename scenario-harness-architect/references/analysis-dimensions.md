## 第一层:业务层

> 回答 **"这个系统到底在解决什么商业问题"**,避免用技术目标替代业务目标。

### 核心四问

| 问题 | 不及格答案 | 及格答案 |
|------|-----------|---------|
| **为谁** | "用户" | "限时活动中抢购商品的 C 端买家 + 设置活动的 B 端运营" |
| **做什么** | "秒杀系统" | "让 C 端在 ≤ 5s 内完成从看到活动到支付成功的完整链路" |
| **成功长什么样** | "能跑起来" | "活动开始 10s 内售罄 99% 库存,支付成功率 ≥ 98%,无超卖" |
| **边界在哪** | 没定义 | "不做商品管理、不做物流、不做售后;只做活动 + 下单 + 预扣" |

### 角色建模

每个关键角色必须给出:

| 字段 | 说明 |
|------|------|
| 角色名 | C 端买家、B 端运营、风控、财务、客服… |
| 核心诉求 | 按重要度排序的 Top 3 |
| 关键动作 | 在系统中最高频的 3–5 个动作 |
| 成功指标 | 这个角色视角下"好用"意味着什么 |
| 失败成本 | 体验崩坏时的业务影响 |

### 核心业务流程

**提炼标准**:

- 每条流程必须是**业务主路径**(用户视角的一次完整任务),不是技术动作
- 每条流程至少给出:触发条件 → 主路径步骤 → 结束态 → 关键指标
- 3–7 条最合适,超过 7 条通常是粒度太细

**反例**:

- ❌ "登录 / 注册 / 查询 / 下单 / 支付"——这是接口列表
- ✅ "买家完成一次秒杀"(含看活动、进入详情、抢购、填地址、支付、查看订单)
- ✅ "运营配置一场活动"(含选品、定价、库存、时段、规则、预热、上线)

### 关键业务指标

每个指标要写清:

- 指标名
- 业务含义
- 目标值 / 警戒值
- 数据来源
- 与技术指标的映射关系(如"支付成功率 → 依赖下游支付系统可用性 + 限流策略")

**常见指标速查**:

| 业务类型 | 典型业务指标 |
|---------|------------|
| 电商/秒杀 | GMV、订单量、支付成功率、售罄率、超卖率、退款率 |
| 社交/IM | DAU、消息发送量、送达率、时延、群活跃度、留存 |
| 支付/清结算 | TPS、资金准确率、对账差错率、清算时效、合规通过率 |
| Feed/内容 | 日均阅读、时长、互动率、曝光/点击、冷启成功率 |
| 推送 | 覆盖率、到达率、点击率、打扰率、退订率 |

### 陷阱清单

| 陷阱 | 修正 |
|------|------|
| 用"功能点"冒充业务流程 | 回到用户视角,问"用户在做一件什么完整的事" |
| 业务指标全是技术指标 | 至少一半必须是商业指标(GMV/留存/转化) |
| non-goals 缺失 | 没有 non-goals 的 MVP 一定会无限蔓延 |
| 角色合并 | 把"买家"和"运营"合成"用户",后面方案必错 |

## 第二层:规模与约束层

> 把业务目标翻译成**硬约束**。方案选型必须被这些数字卡住,否则就是空中楼阁。

### 六类硬约束

| 类别 | 必须给出 |
|------|---------|
| **流量** | 峰值 QPS / 均值 QPS / 读写比 / 热点 key 比例 / 突发形态(尖峰/持续) |
| **数据量** | 存量 / 日增 / 最大单条 / 冷热分布 / 保留期 |
| **延迟** | P50 / P99 / P999 / 端到端上限 |
| **一致性** | 强一致 / 会话一致 / 最终一致,分路径标注 |
| **可用性** | SLA(99.9 / 99.95 / 99.99)/ RTO / RPO / 故障半径 |
| **成本与合规** | 成本上限 / 等保级别 / 审计要求 / 数据主权 / 个保法 |

### 费米估算法(核心技能)

**标准流程**:

```
DAU × 人均关键动作次数 × (活跃时段 / 24h) × 峰谷比 = 峰值 QPS
峰值 QPS × 2~5 = 设计目标(留冗余)
```

**举例:秒杀峰值**

```
假设:
  - 参加活动用户 100w
  - 活动集中在 1 秒钟(典型抢购)
  - 每人平均点击 5 次(重试/误触)

峰值 QPS = 100w × 5 / 1s = 500w/s
设计目标 = 500w × 0.3(限流/削峰)= 150w/s 进系统
真正到达下单 = 150w × 0.01(异步排队 + 防刷)= 1.5w/s TPS
```

这个推导链条每一步都要写在 `scale-constraints.md` 里,agent 后续才能按此做限流/缓存/队列设计。

### 一致性分路径决策

**决策矩阵**:

| 路径类型 | 一致性要求 | 典型手段 |
|---------|----------|---------|
| 资金 / 库存扣减 | 强一致 | DB 事务 + 分布式锁 / Redis Lua |
| 订单主状态 | 强一致 | DB 事务 + 状态机 |
| 订单衍生视图(列表/统计) | 最终一致 | CDC / 消息 + 物化视图 |
| 社交互动(点赞/评论) | 最终一致 | 异步 + 幂等去重 |
| Feed 流 | 最终一致 | 推拉结合 + 容忍短窗口 |
| 搜索 | 最终一致 | 异步索引 + 可接受秒级延迟 |

### SLA 分级参考

| 等级 | 年故障时间 | 典型业务 | 技术代价 |
|------|----------|---------|---------|
| 99.9% | 8.76h | 内部工具、非核心辅助 | 单区域 + 基础冗余 |
| 99.95% | 4.38h | 通用 C 端业务 | 多可用区 + 主备 |
| 99.99% | 52.6min | 核心交易 / 支付 | 同城双活 + 异地灾备 |
| 99.999% | 5.26min | 金融核心 / 电信 | 多活 + 端到端容灾演练 |

**不要随意承诺 99.99%**,每多一个 9 成本和复杂度翻倍以上。

### 陷阱清单

| 陷阱 | 修正 |
|------|------|
| 只说"高并发""海量数据" | 给出具体数字(QPS / TB / ms) |
| 一致性一刀切 | 按路径分级,混合使用 |
| 忘了冷热比 | 冷热分层能极大降成本 |
| SLA 盲目追高 | 99.9% 到 99.99% 差十倍钱 |
| 合规没有具体条目 | 至少点名"资金审计 / 个保 / 等保几级" |

## 第三层:难点与方案对比层

> 难点不等于模块,方案不等于技术名词。**难点 = 场景 × 约束** 组合出的棘手问题;**方案 = 在约束下的 trade-off 决策**。
>
> 📌 **调研驱动原则**:本层是调研证据最重要的消费者。每个难点和方案都应尽量用开源项目 issue、大厂实战、社区踩坑帖来佐证,而不是纯靠常识拍脑袋。

### 难点提炼的 10 条线索

| # | 线索 | 识别问句 |
|---|------|---------|
| 1 | 高并发热点 | 是否存在单 key/单行被全流量打爆的风险? |
| 2 | 资源竞争 | 是否存在库存/名额/余额等必须精确扣减的资源? |
| 3 | 一致性边界 | 哪些路径要强一致?跨服务/跨库如何保证? |
| 4 | 海量读 | 读 QPS 是否远超 DB?用什么缓存链路? |
| 5 | 海量写 | 写是否成为瓶颈?是否需要分库分表/异步化? |
| 6 | 消息可靠性 | 消息必达/去重/顺序/堆积如何保证? |
| 7 | 实时性 | 是否需要长连/推送/流计算?端到端延迟怎么控? |
| 8 | 存储选型 | 结构化/半结构化/时序/图?冷热分层? |
| 9 | 容灾 | 故障隔离、降级、回滚、多活怎么做? |
| 10 | 安全/防刷 | 身份、权限、风控、限流、审计? |

### 难点描述模板

每个难点必须写成:

```
【C{n}】<一句话问题定义>

分类:   [热点 / 一致性 / 海量读 / 海量写 / 可靠性 / 实时 / 安全 / 存储 / 容灾]
约束:   [引用 scale-constraints.md 的具体数字]
调研佐证: [调研:R-XXn] <开源项目/社区帖中的真实案例>
为什么决定性:
  - 若不解决,会发生 <具体业务后果>
  - 这是架构形态的岔路口(选 A 系统形态 vs 选 B 系统形态)
典型误区:
  - 用 <常见错误方案> 会在 <什么时候> 崩溃
初步方向:
  - 候选方向 1、候选方向 2、候选方向 3
```

### 方案矩阵设计方法

**为什么要矩阵**:强迫思考 trade-off,防止"一眼选最熟悉的"。

**矩阵纵轴(候选方案)**:2–3 个,每个必须在现实中可落地,不能凑数。

**矩阵横轴(评估维度)**:

| 维度 | 评估点 |
|------|-------|
| 核心思路 | 一句话概括 |
| 一致性保证 | 强/最终/弱 + 靠什么保证 |
| 吞吐 / 延迟 | 预估数量级 |
| 复杂度 | 实现 / 运维 / 调试 |
| 失败模式 | 最坏情况下系统行为 |
| 适用规模 | 在什么量级以下/以上适用 |
| 成本 | 机器 / 带宽 / 运维人力 |
| 团队门槛 | 现有团队是否 hold 得住 |

**推荐判据**(选方案的内在逻辑):

1. 先排除**违反硬约束**的方案(Gate 1 定下的)
2. 再看**失败模式是否可接受**(最坏情况业务能否承受)
3. 最后看**复杂度与团队匹配度**

不是"最牛的技术",而是"在约束下综合最优"。

### 横切方案(必须单独讨论)

下面这些几乎所有场景都会用到,在 `solutions.md` 里单独开一节:

| 横切方案 | 常见手段 |
|---------|---------|
| 流量削峰 | 异步队列、排队机、令牌桶、漏桶、熔断 |
| 多级缓存 | 本地 → Redis → DB,旁路/写穿/写回 |
| 缓存三连问 | 穿透(布隆)/ 击穿(单飞)/ 雪崩(错峰 + 永不过期的热点) |
| 分布式事务 | 2PC / TCC / Saga / 事务消息 / 本地消息表 |
| 幂等 | 业务唯一键 + 状态机 + 去重表 |
| 限流熔断 | Sentinel / Resilience4j / 网关层 + 服务层双重 |
| 可观测性 | 指标 / 日志 / 链路追踪(RED + USE 方法) |
| 数据对账 | 实时小对账 + 离线总账 + 差错补偿 |

### 深度拆解方法论(solutions.md 之后,为每个难点产出 deep-dive)

> **动机**:方案矩阵回答"选哪个",但 trade-off 的深层逻辑需要**单独的教学文档**才能讲透。一份只有矩阵的 solutions.md 过不了资深架构师的评审——"为什么 A 在 X 规模会崩、为什么 B 能 work"必须有论证。这就是 `deep-dives/C{N}-*.md` 的职责。
>
> **完整模板 + 写作规则 + 自检清单见** `deep-dive-template.md`。下面只讲方法论内核。

#### 深度拆解的 5 个思考引擎

对每个难点做深拆时,必须过完下面这 5 个引擎,缺一不可。它们是产出一份高质量 deep-dive 的 "脚手架"。

| # | 引擎 | 追问方式 | 对应模板节 |
|---|------|---------|-----------|
| 1 | **第一性原理追问** | "为什么难? → 因为 X 约束 → 那 X 为什么存在 → 因为 Y 物理限制 → ..." 追到物理层或业务红线 | §1 问题的本质 |
| 2 | **概念分类学** | "这个领域有哪些必须先定义的词汇?本场景落在哪一侧?" 没有词汇,讨论就是鸡同鸭讲 | §2 关键概念 |
| 3 | **演化叙事** | "从最朴素方案开始,每代如何崩溃?崩溃的物理原因是什么?下一代如何解决?" 至少 3 代 | §3 方案演化 |
| 4 | **二阶反思** | "采用这个方案会**新引入**什么难题?方案本身的边界在哪(什么规模以上不再 work)?" | §4 推荐深拆 |
| 5 | **反模式体检** | "业界有哪些看似合理、实际会崩的方案?分别错在哪?" 帮读者避坑 | §7 反模式 |

任何一个引擎偷懒,deep-dive 就会退化成"换个排版的 solutions.md"。

#### 决策文档 vs 教学文档的分工(硬约束)

| 维度 | `solutions.md`(决策文档) | `deep-dives/C{N}-*.md`(教学文档) |
|------|-------------------------|-----------------------------------|
| 问题 | 选哪个 | 为什么选它 + 每一寸怎么想透 |
| 读者 | 架构评审 / 扫读 | 反复精读 / 传帮带 / agent 背景 |
| 呈现 | 表格为主,A/B/C 并列对比 | 叙事为主,聚焦**单一推荐路径**的深拆 |
| 字数 | 每个难点 300–800 字(矩阵紧凑) | 每份 2500–7000 字 |
| 数字 | 对比项(QPS、复杂度评级) | 论证过程中的硬数字(物理约束、崩溃阈值) |
| 代码 | 通常不含代码 | 必含 ≥ 30 行关键代码/SQL/Lua |
| 演化叙事 | 无 | 必有(≥ 3 阶段) |
| 反模式 | 不体现(只列推荐) | 必列(≥ 3 个) |

**混写警示**:在 `solutions.md` 里塞大段代码和第一性原理推理,或在 `deep-dive` 里列 A/B/C 矩阵——都会让两份文档都变得难读且信息冗余。

#### 深度合格线(金线测试)

一份 deep-dive 交付前做一次 "金线测试":

> **假设一位中级工程师只读这份 deep-dive,他能否在白板做 40 分钟脱稿分享,且回答得了来自资深架构师的 3 个"为什么"追问?**

能,就合格。不能,一定是 5 个引擎中有偷懒的环节,回头补齐。

### 陷阱清单

| 陷阱 | 修正 |
|------|------|
| 把模块当难点 | 问"不解决这个系统会怎样" |
| 方案只有一个 | 至少写两个候选,才能看出 trade-off |
| 只列技术名词 | 每个方案必须讲清机制(怎么做到的) |
| 忽略失败模式 | 最坏情况一定要分析 |
| 横切方案混在业务难点里 | 单独一节,避免每个难点都重复写一遍缓存策略 |
| solutions.md 越写越长试图塞进深度论证 | 停下,把深度论证搬去 `deep-dives/C{N}-*.md`,solutions.md 保持决策矩阵体态 |
| deep-dive 退化成矩阵 | 回到 5 个思考引擎,检查追问链条/演化叙事/二阶反思是否到位 |

## 第四层:架构与实施层

> 把前三层结论落成**可视化架构 + 可并行的实施路径**。agent 读这一层来写代码,模糊一点后面全乱。

### 架构图三件套

| 图 | 类型 | 作用 |
|---|------|------|
| 系统架构图 | Mermaid `graph TB` | 端/网关/业务/存储/基础设施分层,展示全局 |
| 核心流程时序图 | Mermaid `sequenceDiagram` | 至少 2 张,展示主路径上组件协作 |
| 数据模型图 | Mermaid `classDiagram` / 表格 | 核心实体 + 关系 + 关键字段 |

> **关键**:每张图都要有对应的文字解读。图是"how",文字是"why"。

### 模块分解方法

**原则**:

- **单一职责**:一个模块只对一个业务能力负责
- **边界稳定**:模块之间靠契约(接口/事件)通信,内部自由演进
- **与难点对齐**:通常每个核心难点落在 1–2 个模块
- **数量合理**:MVP 阶段 5–15 个模块最佳

**每个模块必须给出**:

| 字段 | 说明 |
|------|------|
| 编号 | M01..Mn |
| 模块名 | 业务语义,不是"service1" |
| 职责 | 一句话 + 3–5 条具体能力 |
| 对应难点 | 关联 challenges.md 的 C{n} |
| 依赖 | 上游 / 下游 / 基础设施 |
| 对外接口 | 主要接口清单(详细签名放 architecture.md) |
| 技术栈 | 语言/框架/库的选型(来自 Gate 2) |
| 数据归属 | 哪些表 / 哪些 key 归这个模块 |

### 数据模型设计要点

- 先画**核心实体关系**,再补扩展字段
- 每个实体必须给:主键策略、分片键(若分库)、索引、关键约束(唯一/非空/FK)
- **一致性边界**:哪些实体在同一事务内、哪些靠消息最终一致,必须标明

### 接口契约设计要点

每个对外接口给出:

- HTTP method + path(或 RPC service + method)
- 请求参数(含校验规则)
- 响应结构 + 错误码
- **幂等性**:是否幂等 / 幂等键是什么
- **限流**:配额 / 速率
- **鉴权**:需要什么身份

### 实施路径分层

**三层切片**:

```
MVP(2–4 周内交付)
  └─ 核心 2–4 个模块,跑通主路径,打通最关键 1 条难点
Phase 2(MVP + 1–3 月)
  └─ 补齐 next N 个模块,覆盖剩余难点,补上容灾/可观测
Phase 3(长期演进)
  └─ 性能深挖、多活、国际化、复杂风控等
```

### 风险矩阵

至少 5 条,每条含:

| 字段 | 说明 |
|------|------|
| 风险描述 | 具体不抽象 |
| 触发条件 | 在什么场景下会发生 |
| 影响级别 | P0 / P1 / P2 |
| 影响范围 | 哪些业务 / 多少用户 |
| 应对策略 | 预防 + 发生后 |
| 演练方式 | 如何验证应对有效 |

### 陷阱清单

| 陷阱 | 修正 |
|------|------|
| 只有系统架构图没有时序图 | 补至少 2 张核心时序 |
| 模块用技术切("api 层""service 层") | 用业务能力切("下单""履约""风控") |
| 接口契约缺幂等/错误码 | 补齐,agent 没这两项写不出正确代码 |
| 实施路径没有验收标准 | 每个 Phase 必须有验收(功能 + 性能) |
| 风险矩阵只写风险不写应对 | 必须带应对 + 演练方式 |

## 四层之间的一致性检查

产出 `architecture.md` 和 `playbook.md` 之前,做一次跨层对齐:

| 检查项 | 方法 |
|-------|------|
| 业务指标 ↔ SLA 对齐 | 业务指标 (如"支付成功率 98%") 是否对应了可用性/延迟 SLA |
| 约束 ↔ 难点对齐 | 每个约束是否都至少触发了一个难点,没被忽略 |
| 难点 ↔ 方案对齐 | 每个难点是否都在 solutions.md 里有对应方案 |
| 方案 ↔ 模块对齐 | 每个方案是否都落到一个或多个模块 |
| 模块 ↔ 实施路径对齐 | MVP 是否覆盖了所有 P0 难点的最小方案 |
| 风险 ↔ 容灾方案对齐 | 风险矩阵中的故障类型是否都有对应容灾/降级 |

这 6 条对齐任何一条不过,回上一层修正,不要强行推进。
