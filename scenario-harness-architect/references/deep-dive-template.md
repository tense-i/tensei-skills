# 深度拆解文档模板(deep-dives/C{N}-*.md)

> 本文档规定 `deep-dives/C{N}-<name>.md` 每一份**单难点深度教学文档**的写作标准。
> 读者:架构师本人反复复盘 / TL 传帮带 / harness prompt 作为 agent 的"背景知识"引用。
> **定位**:教学文档(不是决策 checklist),必须读起来像一位资深架构师在把问题讲透。

---

## 为什么需要这一类文档

- `challenges.md` 只回答"难点是什么",粒度浅
- `solutions.md` 只回答"选哪个",像决策矩阵,扫一眼就忘
- **`deep-dives/C{N}-*.md` 回答"为什么这是个难点、人类是如何把它解出来的、推荐方案的每一寸是怎么想透的"**

对标**大佬写的深度技术博客**(如《如何设计一个秒杀系统》《Designing Data-Intensive Applications》的章节、Stripe/Discord Engineering 的长文),而**不是**PPT 式 bullet 清单。

---

## 强制结构(8 节缺一不可)

每份 deep-dive 文件必须按顺序包含以下 8 节,缺节或合并节都判不及格。

```
# C{N}. {难点名} — 深度拆解

> 本文件定位 / 读者 / 与 challenges/solutions 的关系
> 关联:challenges.md §C{N}、solutions.md §C{N}、research.md §R-XX...

## 1. 问题的本质(First Principles)
## 2. 关键概念与分类学(Taxonomy)
## 3. 方案空间的演化(Evolution)
## 4. 推荐方案的深度拆解(Deep-Dive of Chosen Path)
## 5. 与其他难点的交互(Cross-Challenge Interactions)
## 6. 业界真实案例(Industrial Evidence)
## 7. 常见反模式与误区(Anti-Patterns)
## 8. 工程落地检查清单(Engineering Checklist)
```

---

## 1. 问题的本质(First Principles)

**目标**:回答"为什么这件事**根本上**难",让读者拥有"从物理/数学/架构约束层面看这个问题"的能力。

**写作要求**:

- **追问链条**:像苏格拉底一样,一层一层追问下去
  - 示例(秒杀库存扣减):
    - 为什么库存扣减难? → 同一行被高并发竞争
    - 为什么竞争会导致性能崩? → DB 行锁串行化
    - 为什么必须用锁? → CAS 级别的原子语义需要
    - 能绕开锁吗? → 把状态搬到单线程的 Redis
    - 那为什么 Redis 不用锁? → 单线程天然原子
    - 但 Redis 挂了怎么办? → 这就引出了下一层难题...
- **明确物理边界**:CPU 查表耗时、网络 RTT、单行锁串行、单机内存上限、单分片 QPS 上限等客观约束
- **明确业务边界**:什么红线不能破(0 超卖、支付成功率、SLA)

**反模式**:
- ❌ "这是一个高并发问题,很难" — 没有任何信息
- ❌ 直接列方案不解释为什么问题存在
- ✅ "问题之所以难,是因为 X 物理约束 + Y 业务约束组合,导致 Z 朴素方案在 N QPS 以上必然崩溃"

**篇幅参考**:400–800 字。

---

## 2. 关键概念与分类学(Taxonomy)

**目标**:给读者一套讨论该问题的**词汇表**。没有词汇,所有讨论都是鸡同鸭讲。

**写作要求**:

- 列出该难点领域的 **3–6 个核心概念**,每个概念给出:
  - 定义(一句话)
  - 对立概念或兄弟概念(如 静态数据 vs 动态数据、静态热点 vs 动态热点)
  - 典型举例
  - 在本场景中落在哪一侧
- 必要时画 2×2 矩阵或决策树帮助分类

**秒杀领域示例**:
- 静态数据 vs 动态数据(数据是否随访问者变化)
- 静态热点 vs 动态热点(能否提前预测)
- 下单减库存 vs 付款减库存 vs 预扣库存(减库存时机)
- 应用层排队 vs 数据库层排队(锁等待位置)
- 有损削峰 vs 无损削峰

**反模式**:
- ❌ 只给名词不给定义
- ❌ 定义等同于"XX 就是 XX"(循环定义)
- ✅ 每个概念都能让读者准确判断"当前场景落在哪一侧"

**篇幅参考**:300–700 字。

---

## 3. 方案空间的演化(Evolution of Solution Space)

**目标**:从**最朴素方案**开始,一步步展示人类工程界是**如何把这个问题解出来的**。让读者理解每一代方案解决了什么、又带来了什么新问题。

**写作要求**:

- 至少 **3–5 个演化阶段**(不是 3–5 个并列方案,是**时间/规模维度上递进**的演化)
- 每个阶段必须回答:
  1. 这一代的做法是什么(一句话 + 可能的小图/SQL/伪码)
  2. 它能撑多大规模(具体数字,不是"中等规模")
  3. 为什么会崩(崩的物理原因,不是"不够用")
  4. 下一代如何解决(引出下一阶段)
- 可用一张演化总表收尾

**秒杀"流量消化"演化示例**:

```
阶段 1:单体 Tomcat + MySQL 直写
  → 能撑 ~500 QPS,10w 并发即刻雪崩(DB 连接耗尽 + GC 风暴)

阶段 2:前置 Nginx 静态化 + 读写分离
  → 能撑 ~5k QPS,写瓶颈依然在单行 DB

阶段 3:Redis 预扣 + MQ 异步落单
  → 能撑 ~3w QPS,但 Redis↔DB 一致性成为新难题

阶段 4:分层过滤(CDN → SLB → 网关 → BFF → Redis)
  → 能撑 ~500w QPS 入口,实际写层 ≤ 500 TPS

阶段 5:库存分段 + 多活
  → 单 SKU 突破单分片瓶颈,容灾跨地域
```

**反模式**:
- ❌ 只写最终方案(读者不知道为什么这样)
- ❌ 几个方案并列没有演化关系
- ❌ 不给规模数字("能撑一些")
- ✅ 每个阶段都说"能撑 N QPS,在 M 时会以 X 方式崩溃"

**篇幅参考**:600–1200 字。可含演化对比表。

## 4. 推荐方案的深度拆解(Deep-Dive of Chosen Path)

**目标**:对 `solutions.md §C{N}` 中推荐的那个方案,**把每一寸都讲透**。

**写作要求**(4 个子节缺一不可):

### 4.1 为什么这个方案 Work(第一性原理论证)

- 不是"因为业界都这么用"
- 而是"因为 X 原理 + Y 约束,所以这个方案在 Z 规模下必然可行"
- 举例:"Redis Lua 原子扣减 work,是因为 Redis 单线程执行 Lua 时,整个脚本被视为一条命令,期间无法被其他 client 打断;所以'判断 + 扣减'的组合操作具备原子性。"

### 4.2 核心机制(带关键代码/SQL/配置片段)

- 关键 Lua / SQL / 伪码(30–100 行)
- 关键 key 设计 / 表结构 / topic 规划
- 关键参数选择依据(为什么 TTL 是 24h 不是 1h)
- 不贴整段业务代码,贴**决定正确性/性能的关键片段**

### 4.3 工程陷阱(至少 3–5 个)

- 必须是"踩过的坑",不是"理论上可能"
- 每个陷阱给出:症状 → 原因 → 修复/规避
- 示例:
  - 陷阱:Redis Cluster 模式下 Lua 脚本跨 slot 执行报错 → 原因:`stock` 和 `history` 两个 key 不在同一分片 → 修复:用 hash tag `{activityId}:{skuId}` 绑定
  - 陷阱:订单取消盲目回补库存 → 原因:没先查幂等表 → 修复:先 `HGET hot_deduction_history` 验证确实扣过再回

### 4.4 边界条件与二阶问题

- 这个方案在什么规模以上/以下**不适用**(如:单 SKU 库存 > 10k 就必须分段)
- 采用这个方案本身会引入什么**新难题**(如:上了 MQ 异步后,Redis↔DB 一致性成了新难题)
- 这些新难题如何应对(引到下一个难点或横切方案)

**反模式**:
- ❌ 只有 Lua 代码,没有"为什么这么写"
- ❌ 只讲 happy path,不讲陷阱
- ❌ 抄 solutions.md 的 trade-off 表再贴一次(重复)
- ✅ 读者读完能自己给别人讲清楚"这个方案为什么这样设计"

**篇幅参考**:1000–2000 字。这一节通常是全文最长的。

## 5. 与其他难点的交互(Cross-Challenge Interactions)

**目标**:阐明这个难点的方案如何**传染**到其他难点,避免读者孤立看待每个难点。

**写作要求**:

- 至少列出 2–3 条与其他 C{x} 的因果/依赖关系
- 每条说清:
  - 方向:这个方案影响 C{x},还是 C{x} 的方案约束了这里
  - 机制:具体怎么相互作用
  - 后果:组合在一起带来的新约束

**秒杀 C1(库存扣减)与其他难点的交互示例**:
- C1 → C2(Redis↔DB 一致性):Redis 扣减成功但 MQ 失败就成了"少卖",必须配对账
- C1 → C7(单 SKU 瓶颈):单 Redis key 的 QPS 上限直接决定单 SKU 的抢购峰值
- C5(风控) → C1:风控判定在扣减**之前**(如果之后,库存已经扣出去了)

**反模式**:
- ❌ "C1 和 C2 有关系"(没说怎么有关系)
- ❌ 只写下游不写上游
- ✅ 能画出难点之间的依赖图,并解释每条边

**篇幅参考**:300–600 字。可配一张小的依赖图。

## 6. 业界真实案例(Industrial Evidence)

**目标**:把 `research.md` 中的真实案例**落到这个难点上**,让方案有"来自实战"的信度。

**写作要求**:

- 至少 2 个案例,格式:
  ```
  ### 案例 X:{公司/项目名} 在 {场景规模} 下如何解决 C{N}
  - 来源:[R-BT1] / [R-OS2] / [R-CM1](引用 research.md)
  - 场景规模:DAU / QPS / 数据量
  - 核心做法:一段话(带关键数字)
  - 关键 trade-off:选了什么放弃了什么
  - 翻车记录(如有):遇到过什么问题,如何修复
  - 对本方案的启示:可借鉴/可避免的点
  ```
- 尽量包含**不同规模**的案例(小厂/中厂/大厂各一),说明方案的适用边界
- 如 research.md 中没有对应案例,**必须**在 Step R 补充调研,不允许"查不到就不写"

**反模式**:
- ❌ 泛泛"某大厂用 Redis"(无引用、无数字、无 trade-off)
- ❌ 只引一个案例,不体现规模差异
- ✅ 读者能明确判断"在我这规模下,这个方案验证过了"

**篇幅参考**:400–800 字。

## 7. 常见反模式与误区(Anti-Patterns)

**目标**:帮读者**避免**业界常见的错误方案——这些方案看起来合理,实际会崩。

**写作要求**:

- 至少 3–5 个反模式,格式:
  ```
  ### 反模式 X:{方案名}
  - 看起来合理的理由:...
  - 实际为什么错:...(根因)
  - 会在什么场景爆炸:...
  - 正确做法:引用本文第 4 节的推荐方案
  ```
- 反模式要具体,不是"设计不合理"

**秒杀示例反模式**:
- "用分布式锁替代 Redis Lua":性能损失 ≥ 20%,没有额外安全收益
- "用 SELECT FOR UPDATE 扛秒杀":单行锁会把 QPS 锁死到 100 级
- "先查余量再扣减(两条命令)":非原子,必定超卖
- "异步订单 + 同步风控":风控延迟毁掉抢购体验
- "库存减为负再补":数据完整性崩坏,无法对账

**反模式**:
- ❌ 把所有非推荐方案都当反模式(应该只列"常见的致命错")
- ✅ 每个反模式都能让读者一眼看出"原来这就是那个坑"

**篇幅参考**:400–800 字。

## 8. 工程落地检查清单(Engineering Checklist)

**目标**:给实施这个方案的工程师一份**动手 checklist**,确保关键点不漏。

**写作要求**:

- 5–15 条可勾选项,按"设计 → 实现 → 测试 → 上线 → 运营"顺序组织
- 每条要具体,不是"设计要合理"

**秒杀 C1 示例**:

```
□ 设计:stock key 与 history key 使用同一 hash tag
□ 设计:idempotencyKey 在客户端生成并贯穿 BFF → Redis → MQ → DB
□ 实现:Lua 脚本先 HGET 幂等表,存在直接返回历史结果
□ 实现:history 表条目设置 24h TTL(防内存爆炸)
□ 实现:回补库存前必须先查幂等表,不存在则拒绝回补
□ 测试:压测单分片 5k QPS 30 分钟,观察内存与 P99
□ 测试:混沌演练 Redis 主挂,验证降级路径
□ 测试:回滚 + 重试 + 乱序组合场景不超卖
□ 上线:Kill-switch 到位,10s 内可关停
□ 运营:每 30s 对账一次,差错立即 P1 告警
```

**反模式**:
- ❌ 过于抽象("实现要幂等")
- ✅ 每条都能在 CI / 压测 / code review 中勾选

**篇幅参考**:200–500 字。

## 全文写作规则(强制)

### 语言风格

- **教学式叙事**,不是决策 checklist:用"我们"/"读者"视角,不是冷冰冰的参数表
- **中文为主,技术术语中英对照**
- **长句比短句好一点**:不要全文都是 bullet。用 3:7 左右的比例搭配段落与列表
- **段落有连接词**:"所以…""那么…""但是…""注意…" — 保持阅读流

### 必须有的元素

| 元素 | 要求 |
|------|------|
| **数字** | 每节至少出现 2 个具体数字(QPS / ms / 字节 / 百分比) |
| **代码/SQL/Lua** | 第 4 节至少 30 行以上的代码或伪码 |
| **图** | 至少 1 张 Mermaid(时序/架构/状态机/流程),在第 3、4、5 节中任选一处 |
| **研究引用** | 至少 3 处 `[R-XX]` 引用 research.md |
| **其他文档引用** | 至少引用一次 challenges.md 和 solutions.md,避免重复内容 |

### 必须没有的

- ❌ 复制 solutions.md 的 trade-off 表(这里不是决策文档)
- ❌ 把章节写成"方案 A、方案 B、方案 C"的并列矩阵(那是 solutions.md 的事)
- ❌ 整章节只有一个大列表
- ❌ 使用"合理""高效""可靠"这类 agent 无法判定的词
- ❌ 无数据的断言("性能很好")
- ❌ 罗列名词不解释(只说"用 Kafka"而不解释"为什么 Kafka 而不是 RabbitMQ")

### 总篇幅参考

| 复杂度 | 单份 deep-dive 字数 |
|--------|-------------------|
| 简单难点(C 类偏工程) | 2500–3500 字 |
| 中等难点(多数 C) | 3500–5000 字 |
| 复杂难点(核心 C,如库存扣减、一致性) | 5000–7000 字 |

过短意味着深度不够(基本等于把 solutions.md 复述一遍),过长意味着掺杂了不属于这个难点的内容。

## 引用与跨文档关系

deep-dive 与其他文档的关系:

```
challenges.md §C{N}        — 定义问题(2–4 段)
     ↓ (引用)
deep-dives/C{N}-*.md       — 教学展开(3000–7000 字)
     ↓ (引用)
solutions.md §C{N}         — 决策矩阵 + 推荐(1–2 屏)
     ↓ (引用)
architecture.md            — 方案落到具体模块/接口
     ↓ (引用)
harness/modules/*          — 模块 prompt
     ↓ (引用)
harness/tasks/*            — 任务 prompt
```

- **challenges.md** 用 "详见 deep-dives/C{N}-*.md" 带出深度
- **solutions.md** 推荐方案后附 "决策背后的完整推理见 deep-dives/C{N}-*.md"
- **harness/master-prompt.md** 在描述项目背景时引用 deep-dives 作为 agent 的"深度参考知识",但不内联全文
- **harness/modules/** 在描述模块职责时用 "见 deep-dives/C{N}-*.md §4" 精确锚定

**禁止**:
- 在 deep-dive 里粘贴整段 solutions.md / architecture.md 内容
- 在 harness prompt 里粘贴整段 deep-dive 内容(用引用)

## 作者自检清单(写完交付前必过)

下面每一条不过就回炉重写,不要硬交付。

```
□ 8 节全部到位,没有合并/遗漏
□ 第 1 节"问题的本质"有明确的追问链条,不只是"难"
□ 第 2 节给出了 3–6 个核心概念的定义 + 分类
□ 第 3 节至少 3 个演化阶段,每个阶段有规模数字 + 崩溃机制
□ 第 4 节"为什么 Work"的论证不依赖"业界都这么做"
□ 第 4 节有 30 行以上的关键代码/SQL/Lua
□ 第 4 节至少 3 个工程陷阱,每个有"症状 → 原因 → 修复"
□ 第 5 节至少 2–3 条跨难点交互,每条有机制说明
□ 第 6 节至少 2 个真实案例 + research.md 引用
□ 第 7 节至少 3 个反模式,每个有"看似合理的理由"+ "实际为什么错"
□ 第 8 节是可勾选的具体 checklist(≥ 5 条)
□ 全文 ≥ 2500 字
□ 至少 1 张 Mermaid 图
□ 至少 3 处 [R-XX] 引用
□ 没有"合理/高效/可靠"这类模糊词
□ 读者(假想 TL)能据此在白板给团队讲 40 分钟
```

最后一条是最重要的**验收金线**:如果读完这份文档,一位中级工程师能在白板前就这个难点做 40 分钟脱稿分享,那深度就够了;否则不够。
