# Harness Prompt 设计哲学

> 本文档回答一个根本问题:**如何把架构师的决策压进 prompt,让 coding agent 把系统正确地写出来?**
> 这是本 skill 区别于其他架构分析 skill 的核心增量——不仅"想清楚",更要"让 agent 能做出来"。

## Harness Engineering 的核心理念

### 从"写代码"到"写让 AI 写代码的脚手架"

| 传统开发 | Harness 开发 |
|---------|-------------|
| 程序员关心"代码怎么写" | 架构师关心"如何把约束传递给 agent" |
| 代码是交付物 | prompt 包是交付物,代码是副产物 |
| 迭代靠改代码 | 迭代靠改 prompt(或修改 agent 执行的 harness) |
| 知识沉在代码里 | 知识沉在 prompt 里,可复用到下一个场景 |

### Harness 的四个支柱

```
            ┌─────────────────────────┐
            │ Judgment(人类判断)     │  问题定义/不变量/取舍/失败边界
            └───────────┬─────────────┘
                        │
            ┌───────────▼─────────────┐
            │ Context(上下文)        │  业务/约束/证据/架构
            └───────────┬─────────────┘
                        │
            ┌───────────▼─────────────┐
            │ Contract(契约)         │  输入/输出/副作用/不变量
            └───────────┬─────────────┘
                        │
            ┌───────────▼─────────────┐
            │ Verifier(验证器)       │  DoD/测试/压测/代码审查标准
            └─────────────────────────┘
```

- **Judgment 不够**:agent 会把 AI 建议当最终决策,人类也会失去判断训练
- **Context 不够**:agent 不知道"为什么"这么做,会乱发挥
- **Contract 不够**:agent 产出的模块无法拼装
- **Verifier 不够**:agent 说"完成了",但系统跑不通

## 三层 prompt 的职责边界

### 为什么分三层

一个超大 prompt 扔给 agent,会撞到三个墙:

1. **上下文窗口墙**:大项目几万行指令装不下
2. **注意力墙**:agent 会丢前忘后,细节被覆盖
3. **调试墙**:哪个指令错了无从定位

分层后,agent 按需加载:**master 给方向,module 给边界,task 给执行点**。

### master-prompt.md(总控)

**读者**:agent 启动时的第一份指令。

**回答的问题**:
- 我要做什么项目?
- 有哪些全局约束?
- 目录规约是什么?
- 有哪些模块?执行顺序是什么?
- 何时算整个项目"完成"?

**必须包含**:

| 段落 | 内容 |
|------|------|
| 项目定位 | 1 段话,引用 `business.md` |
| 人类已确认判断 | 成功语义、不变量、关键取舍、失败边界,引用 `judgment.md` |
| 全局约束 | 语言 / 框架 / 数据库 / 消息 / 代码风格,引用 `assumptions.md` |
| 整体架构 | 粘贴架构图 or 引用 `architecture.md` |
| 目录规约 | 完整目录树,agent 按这个结构创建文件 |
| 模块清单与执行顺序 | 按拓扑排序(依赖在前) |
| 质量门 | 单测覆盖率、linter、压测要求、文档要求 |
| 子 prompt 索引 | 指向 `modules/*.md` 与 `tasks/*.md` |
| 启动指令 | "请按顺序读取 modules/M01-xxx.md 开始执行" |

**不应该包含**:
- 具体接口签名(放 module/task)
- 具体算法实现(放 task)
- 细节的错误处理逻辑(放 task)

**尺度**:400–800 行。超过说明装了太多细节,应该下沉到 module。

### modules/M{N}-<name>.md(模块级)

**读者**:agent 处理某一个模块时的指令。

**回答的问题**:
- 这个模块的职责边界是什么?
- 它依赖谁?谁依赖它?
- 它对外提供什么接口?
- 它内部分几个子能力?对应哪些 task?
- 完成的验收标准是什么?

**必须包含**:

| 段落 | 内容 |
|------|------|
| 模块编号与名称 | `M01 - 秒杀下单` |
| 对应难点 | 引用 `challenges.md` 的 C{n} |
| 职责 | 一句话 + 3–5 条能力项 |
| 边界 | 明确**不做什么** |
| 上游依赖 | 来源模块 / 外部服务 |
| 下游消费者 | 谁会调这个模块 |
| 对外接口契约 | 方法清单 + 签名 + 错误码 + 幂等性 |
| 内部组件拆分 | 这个模块里还有哪些内聚的子结构 |
| 数据归属 | 哪些表 / 哪些 Redis key 属于这个模块 |
| 技术选型提示 | 库 / 模式(如"用状态机管理订单状态") |
| 任务索引 | 指向 `tasks/M{N}-T*.md` |
| 模块验收标准 | 功能 + 性能 + 一致性 + 可观测性 |

**尺度**:150–400 行。

### tasks/M{N}-T{M}-<name>.md(任务级)

**读者**:agent 真正动手写代码时的指令。

**回答的问题**:
- 具体要实现什么?
- 输入是什么?输出是什么?
- 边界/异常怎么处理?
- 做到什么程度算"完成"?

**必须包含**:

| 段落 | 内容 |
|------|------|
| 任务编号与名称 | `M01-T03 - 库存预扣接口` |
| 所属模块 | `M01` |
| 任务目标 | 一句话 |
| 前置条件 | 已有什么 / 还需什么(若未就绪,指向依赖 task) |
| 输入契约 | 参数 / 格式 / 校验规则 |
| 输出契约 | 返回值 / 错误码 / 副作用(DB 写 / 消息发 / 缓存更新) |
| 实现要点 | 3–6 条,给算法/模式/关键步骤,不给完整代码 |
| 边界与异常 | 所有需要处理的异常路径(超时/并发/重复/回滚) |
| 幂等性约束 | 幂等键是什么,如何保证 |
| Definition of Done | 可判定的验收条目 |
| 最小测试清单 | agent 必须产出的测试用例(至少 5 条) |

**尺度**:80–200 行。

**为什么不直接给代码**:
- 给代码就退化成"写代码",不是 harness
- agent 在不同场景下可能需要微调实现细节
- 代码在 task 里等于锁死,降低 agent 的发挥空间
- 给"要点 + 契约 + 验证",让 agent 自己生成代码,这才是 harness 的魅力

## 五条质量原则

### 1. Self-contained 但引用原文

**坏的做法**:每个 module prompt 都把业务背景复述一遍(膨胀、容易不同步)。

**好的做法**:关键决策要点直接写,背景性/推理性内容用引用:

```markdown
## 业务背景
本模块是秒杀系统的库存预扣核心,详见 `business.md` §2(秒杀主路径)。

## 关键约束(来自 Gate 1)
- 峰值 QPS:1.5w TPS 进入预扣链路
- 一致性:强一致(不允许超卖)
- 延迟:P99 ≤ 50ms
```

这样 agent 能读取关键数字做决策,需要更多背景时去读原文。

### 2. 可判定(Avoid Squishy Language)

**禁用**:"合理""优雅""友好""高性能""可扩展"——agent 不知道怎么判断达到了。

**改成**:

| 模糊 | 可判定 |
|------|-------|
| "高性能" | "P99 ≤ 50ms, 压测 2w QPS 稳定" |
| "优雅的错误处理" | "所有异常路径都有日志 + 指标 + 用户友好错误码" |
| "代码清晰" | "函数 ≤ 50 行,圈复杂度 ≤ 10,无重复代码(sonar rule)" |
| "一致性好" | "所有写路径经过 DB 事务或 Saga 补偿,补偿覆盖所有失败点" |

### 3. 契约优先(Contract-First)

在写任何 task 之前,先确定:

- 输入的结构(schema)
- 输出的结构(schema)
- 副作用(哪些表 / 哪些消息主题 / 哪些缓存 key)
- 失败时回滚什么

**契约先定,实现后写**——这样不同 task 可以并行交给不同 agent 实例执行而不冲突。

### 4. 测试驱动 DoD(Definition of Done 带测试)

每个 task 的 DoD 必须包含:

- 至少 N 条测试用例(正常 / 边界 / 异常 / 幂等 / 并发)
- 指定测试框架(如 `go test` / `jest` / `pytest`)
- 指定覆盖率最低门槛

**举例**:

```markdown
### Definition of Done

1. 接口实现完毕,返回符合输出契约
2. 单元测试覆盖:
   - 正常下单:1 条
   - 库存不足:1 条
   - 重复提交(幂等):1 条
   - 并发扣减(100 协程抢 10 个库存,必须恰好 10 个成功):1 条
   - 依赖故障降级:1 条
3. 压测脚本产出:k6 脚本 `bench/order_create.js`,2w QPS 下 P99 ≤ 50ms
4. 代码覆盖率 ≥ 80%
5. lint 通过,无 TODO/FIXME 残留
```

### 5. 引用而非复述

- harness 与分析文档**不要大段重复**
- 关键数据/决策可以摘要重复(重复几个字没关系),大段背景用引用
- 引用格式:`见 solutions.md §C3/方案B`

## 常见反模式

### 反模式 1:把代码当 prompt

```
写一个 OrderService,里面有 createOrder 方法,参数是 userId, productId, quantity...
```

这是代码,不是 harness。改成契约 + 要点:

```
## 任务:创建订单(下单核心路径)

### 输入契约
- userId (int64, > 0)
- productId (int64, 存在于活动商品表)
- quantity (int32, 1..100)
- idempotencyKey (string, UUID, 24h 内去重)

### 输出契约
- 成功: orderId (int64) + status="pending_payment"
- 失败错误码:INSUFFICIENT_STOCK / DUPLICATE / INVALID_PARAM / ...

### 实现要点
1. 幂等检查:以 idempotencyKey 为 key 查 Redis,命中直接返回
2. 活动校验:商品在活动时段、未下架
3. 库存预扣:调用 M02 的 `deductStock`,非预扣成功立即 INSUFFICIENT_STOCK
4. 订单落库:DB 事务 insert,失败回滚预扣
5. 订单入队:发送 `order.created` 事件到 Kafka(异步后续流程)
```

### 反模式 2:过度描述

一个 task 写 500 行,把背景/理由/历史全写进去。agent 读不完,抓不到重点。

规则:task 超过 200 行,拆成 2 个 task 或把背景下沉到 module。

### 反模式 3:不写 DoD

agent 说"我完成了",然后代码跑不通/测试不过。DoD 是唯一能让 agent 自检的标尺。

### 反模式 4:模块之间 prompt 重复

5 个模块都复述一遍业务背景,每个都讲一遍缓存策略。

规则:通用内容写进 master-prompt,module 只讲自己独特的部分。

### 反模式 5:把架构决策藏在 task 里

比如"用 Redis Lua 脚本做预扣"这种架构决策写在 task 实现要点里,其他 task 不知道。

规则:架构级决策进 module(甚至 master),task 只讲具体实现要点。

### 反模式 6:隐式依赖

Task B 依赖 Task A 的产物,但 Task B 的 prompt 里没写清楚。agent 可能先做 B。

规则:每个 task 的"前置条件"里必须列明依赖哪些 task,以及依赖产物的路径。

## Agent 差异适配

不同 coding agent 有不同偏好,在 master-prompt 里做适配:

### Claude Code

- 擅长:长上下文、多文件协同、工具使用(bash / grep / edit)
- 偏好:
  - 明确的目录规约(创建前自己列 tree)
  - 允许它主动探索代码库
  - 用"先计划再执行"的模式
- 建议 master-prompt 加:
  ```
  请先用 tree 命令展示你将要创建的目录结构,等我确认后再开始。
  每完成一个模块后,运行 `go test ./...` 确认没有破坏。
  ```

### Codex / OpenAI agent

- 擅长:按结构化指令执行
- 偏好:
  - 步骤编号清晰
  - 每步产出可验证
  - 减少开放式"你觉得"问题
- 建议 master-prompt 加:
  ```
  按 Phase 1 → Phase 2 顺序执行,每个 Phase 完成后输出产物清单。
  所有接口必须先写契约(openapi.yaml),再写实现。
  ```

### Cursor / IDE-integrated

- 擅长:针对单文件的精细修改
- 偏好:
  - 明确指定要改哪个文件
  - 指令粒度细
- 建议 task prompt 加:
  ```
  修改文件:src/order/service.go
  在 CreateOrder 函数中插入库存预扣逻辑,保持原有错误码不变
  ```

### Generic / 未指定

默认按 Claude Code 的风格写,兼容性最好。

## Prompt 质量闸门

在交付 harness 之前,对每份 prompt 做以下自检:

### master-prompt 自检

| 项 | 标准 |
|---|------|
| 项目定位清晰 | 读完第一段能说出"这个系统做什么" |
| 人类判断完整 | `judgment.md` 中成功语义、不变量、取舍、失败边界、验收标准均已确认 |
| 全局约束完整 | 语言/库/数据库/消息/代码风格 5 项齐全 |
| 目录规约可执行 | agent 能直接 `mkdir -p` 创建 |
| 模块顺序正确 | 被依赖的在前 |
| 质量门可判定 | 覆盖率/延迟/lint 都有具体数字 |
| 索引完整 | 所有 modules 和 tasks 都能被引用到 |

### module-prompt 自检

| 项 | 标准 |
|---|------|
| 职责单一 | 一句话能说清,不含"顺便做了 X" |
| 边界明确 | 写了"本模块不做什么" |
| 契约完整 | 输入/输出/错误/幂等都写了 |
| 任务拆分合理 | 任务数与 Gate 3 粒度决策一致 |
| 验收标准可判定 | 没有"合理""良好"类词 |

### task-prompt 自检

| 项 | 标准 |
|---|------|
| 任务目标清晰 | 一句话 |
| 输入/输出有 schema | 字段/类型/约束都写了 |
| 实现要点非代码 | 3–6 条方向性指导,不是完整代码 |
| 异常路径齐全 | 超时/重复/并发/依赖故障都覆盖 |
| DoD 可判定 | 测试用例 + 覆盖率 + 性能指标 |
| 依赖关系显式 | 前置 task 写了编号 + 产物路径 |

### 全局自检

| 项 | 标准 |
|---|------|
| 分层无越权 | master 不讲实现,task 不讲架构决策 |
| 无大段重复 | 相同内容不在不同层重复,用引用 |
| 引用有效 | 所有 `见 xxx.md §y` 都真实存在,尤其是 `judgment.md` 中的关键不变量和取舍 |
| Agent 友好 | 把 master-prompt 丢给目标 agent,模拟执行 10 分钟看是否卡住 |

任何一项不过,**回炉重写对应 prompt,不要交付**。
