# 人类判断文档模板 judgment.md

> 目标:把“AI 给出的方案”转化为“人类真正拥有的判断”。
> 本文档不是给 coding agent 看的第一手实现说明,而是给架构师/TL/学习者看的判断训练记录。
> 任何影响系统形态、长期能力和工程风险的关键选择,都必须先进入本文件,再进入 architecture.md 与 harness/*。

---

## 使用协议

1. AI 可以预填候选答案,但必须标注 `[AI建议]`。
2. 需要用户拍板的条目必须标注 `[人类判断]`。
3. 用户确认后改为 `[已确认:用户]` 或 `[已确认:采用默认]`。
4. 不允许用“合理、可靠、高性能、可扩展”替代具体判断;必须写清数字、边界、代价、失败模式。
5. 如果用户尚未确认,后续 `architecture.md` 与 `harness/*` 只能引用该条为“待确认假设”,不能伪装成最终结论。

## 2. 核心不变量:哪些东西绝不能被破坏

| 编号 | 不变量 | 违反后果 | 证明方式 | 状态 |
|---|---|---|---|---|
| I1 | {例如 confirmed_order_quantity <= total_stock} | {资损/体验/合规} | {并发测试/唯一约束/对账} | [人类判断] |

### 不变量自检问题

- 这些不变量是否覆盖了资金、库存、权限、幂等、消息可靠性?
- 哪些不变量必须由数据库/事务兜底,不能只靠代码判断?
- 如果 AI 生成的代码违反这些不变量,测试能否立刻暴露?

## 场景判断题库

### 秒杀 / 库存 / 高并发

- “抢购成功”到底表示资格成功、订单成功还是支付成功?
- 库存以 Redis 为准、DB 为准,还是分阶段各自有不同语义?
- 如何证明不会超卖?证明依赖代码、Redis 原子性、DB 约束还是对账?
- Redis 扣减成功但订单落库失败怎么办?
- MQ 重复投递、ACK 失败、消费者重启后会发生什么?
- 一人一单由哪个层保证?接口、Redis、DB 唯一约束是否都有兜底?

### IM / 消息

- “消息已发送”表示写入服务端、投递到对端、对端已读,还是多端同步完成?
- 消息顺序是全局顺序、会话内顺序、单发送者顺序,还是弱顺序?
- 离线消息保留多久?谁来清理?如何补拉?
- ACK 丢失和重复 ACK 如何处理?

### 支付 / 清结算

- 哪些状态变化必须强一致?哪些报表可以最终一致?
- 幂等键来自哪里?请求号、业务单号、支付流水号还是组合键?
- 对账差错怎么分级?自动补偿还是人工处理?
- 资金类失败是否允许自动重试?重试边界是什么?

### Feed / 内容社区

- 读优化还是写优化优先?为什么?
- 推模式、拉模式、推拉结合分别在哪个规模下崩?
- 删除、屏蔽、隐私变更如何影响已扩散内容?
- 排序结果的“正确”如何验证?

## 质量闸门

交付前检查:

| 检查项 | 合格标准 |
|---|---|
| 问题定义 | 用户能用自己的话复述,不是复制 AI 原文 |
| 不变量 | 至少 5 条,且每条都有违反后果与证明方式 |
| 关键取舍 | 每个 P0 决策都写清选择、放弃、代价、反证条件 |
| 失败模式 | 覆盖重试、超时、重复、部分成功、服务重启、依赖故障 |
| 验收标准 | 每条 P0 不变量至少对应 1 个自动化测试或压测 |
| AI 边界 | 明确哪些任务只允许 AI 起草、不能代替人决策 |
| 学习复盘 | 至少 3 条“下次我会先问的问题” |
