# 场景分析工作流(半自动 + 3 个 Gate)

> 本文档是 `SKILL.md` 中 Workflow 章节的展开版。
> 目标:用**固定的 8+R+J 步流程 + 3 个关键 Gate** 保证产出既稳定、贴合业务意图,又能训练人类判断力,避免被 AI 的流畅输出牵着走。

---

## 模式定位:为什么是"半自动 + 关键点确认"

三种模式对比:

| 模式 | 特点 | 风险 |
|------|------|------|
| Socratic 深度访谈 | 每一步都问用户 | 交互成本高,用户容易疲劳放弃 |
| **半自动 + Gate** | skill 预填默认值,只在分歧点确认 | **贴合度与效率平衡最好(本 skill 采用)** |
| 全自动一键成文 | 输入场景名直接出全套 | 业务贴合度差,关键决策错得离谱 |

**半自动 + Gate 的核心原则**:
- 常识类决策 → skill 自己拍板,标 `[推断]`
- 可从公开资料获取的 → 先调研再分析,标 `[调研:来源名/URL]`
- **影响全局架构的决策** → 必须过 Gate 让用户拍板
- **每个 Gate 有兜底默认值**,用户可以一句"全部采用默认"快速推进
- **关键判断必须先进入 `judgment.md`**,AI 只能给候选,不能把候选伪装成人类已拥有的理解

## Step 0 — 场景元数据与默认假设

**目标**:把场景题转成结构化元数据,建立全流程的共享上下文。

**操作**:
1. 解析用户输入,抽取场景名、业务背景(若提供)、规模线索、技术倾向
2. 生成场景 ID(如 `seckill-v1` / `im-basic`),决定 `output_root`
3. 依据常识为以下要素预填默认值并写入 `assumptions.md`:

| 要素 | 默认推断策略 |
|------|------------|
| 业务类型 | 从场景名匹配(电商/社交/IM/支付/内容/推荐) |
| 目标规模 | 未说明时用行业常见中等规模(见下表) |
| 一致性倾向 | 金融类→强一致;社交类→最终一致;电商类→混合 |
| 可用性目标 | 默认 99.95%,金融/支付 99.99% |
| 合规 | 金融→等保二级 + 资金审计;社交→内容合规 + 数据隐私 |
| 部署形态 | 默认云上容器化,区域级部署 |

**常见场景默认规模速查**:

| 场景 | 默认 DAU/QPS/存量数据 |
|------|----------------------|
| 秒杀 | 单场峰值 10w QPS,后端下单 1–5w TPS,数据量单场百万级 |
| IM | DAU 1000w,在线 200w,消息量日 10 亿,存量 100TB |
| 支付清结算 | TPS 5000,日交易 1 亿,资金流水存量 PB 级 |
| Feed 朋友圈 | DAU 1 亿,读 QPS 百万,写 QPS 10w |
| 推送 | 长连接 1000w,推送峰值 10w/s |
| 订单履约 | 日单量 1000w,峰值 1w TPS |

4. 把预填的假设显式告诉用户,并过 Gate 1 前不再回头

**产出**:`scenarios/<scenario_id>/assumptions.md`(v0 草稿,后续 Gate 会更新)

## Step R — 联网调研

**目标**:在正式分析之前,用联网搜索建立**证据基础**,让后续所有分析建立在真实案例与已验证方案之上,而非纯靠常识推断。

**触发时机**:Step 0 完成后,Step 1 之前。调研结果贯穿后续所有步骤。

### 搜索执行流程

```
1. 从 Step 0 的场景名 + 关键词,生成 3–5 条并行搜索查询
   ├── 查询 A: "{scenario} open source github"(找开源项目)
   ├── 查询 B: "{scenario} 架构 技术方案"(中文大厂博客)
   ├── 查询 C: "{scenario} architecture design scalable"(英文技术文章)
   ├── 查询 D: "{scenario} 踩坑 实战 经验"(社区实战帖)
   └── 查询 E: "{scenario} benchmark RFC specification"(学术/标准,按需)
2. 对每条查询的搜索结果,逐条评估是否为高质量来源
3. 对高质量来源,使用 fetch 抓取详情,提炼关键信息
4. 汇总进 research.md
```

### 高质量来源判定标准

| 来源类型 | 通过条件 | 排除条件 |
|---------|---------|----------|
| 开源项目 | Stars > 500,近 1 年有提交,有 README + 架构文档 | 归档项目、无文档、纯 Demo |
| 大厂博客 | 来自知名工程团队,有具体数据/代码/架构图 | 营销软文、纯概念无实操 |
| 社区帖 | 有实现细节 + 实际数据 + 踩坑经验 | 泛泛而谈、抄袭聚合文 |
| 学术/标准 | 被广泛引用的 RFC/论文/基准报告 | 过时标准(> 5年且已有替代) |

### 信息提炼模板

**开源项目**:
```
- 项目: {name} ({url})
- Stars: {N} | 语言: {lang} | 最近更新: {date}
- 核心架构: {一句话概述}
- 与本场景的关联: {哪些子问题可以参考}
- 关键 trade-off: {该项目选择了什么,放弃了什么}
```

**大厂方案**:
```
- 来源: {company} - {title} ({url})
- 场景规模: {DAU/QPS/数据量}
- 核心方案: {一句话概述}
- 关键 trade-off: {选择了什么,为什么}
- 可借鉴点: {对本场景有价值的具体做法}
```

**社区洞察**:
```
- 来源: {platform} - {title} ({url})
- 核心观点: {一句话}
- 实战数据: {如有}
- 可信度: 高/中/低(依据: {理由})
```

### 调研汇总要求

调研完成后,在 `research.md` 末尾必须产出:

1. **关键发现**(3–5 条):跨多个独立来源佐证的共识结论
   - 格式:`[共识] {结论} — 来源: {来源1}, {来源2}, {来源3}`
2. **争议点**(0–3 条):不同来源给出矛盾方案的地方
   - 格式:`[争议] {问题} — A方: {观点+来源}, B方: {观点+来源}`
   - 争议点自动进入最近的 Gate 让用户决策
3. **调研覆盖度自检**:对照 SKILL.md 的证据门槛,确认来源数量达标

### 调研结果如何影响后续步骤

调研不是独立产物,而是**贯穿后续所有分析的证据底座**:

- **Step 1 business.md**:参考开源项目 README 和大厂博客的业务描述,校准业务本质
- **Step 2 scale-constraints.md**:用大厂博客中的真实规模数据替代纯费米估算
- **Step 3 challenges.md**:从开源 issue + 社区踩坑帖中提炼真实难点,用 `[调研:URL]` 标注
- **Step 4 solutions.md**:方案矩阵必须覆盖开源项目已验证的方案,引用 `research.md §{section}`
- **Step 5 architecture.md**:参考成熟开源项目的模块分解,避免重新发明轮子
- **Step 7 harness/**:master-prompt 中引用开源项目作为 agent 的参考实现

**产出**:`research.md`

## Step 1 — 业务层分析

**目标**:抽离业务本质,避免陷入"为了技术而技术"。

> 📌 本步骤应参考 `research.md` 中开源项目的 README 和大厂博客的业务描述来校准分析。

**四问框架**:

1. **为谁**:核心用户角色是谁?各角色的核心诉求?
2. **做什么**:系统在每个角色视角下完成什么事?
3. **成功长什么样**:关键业务指标 (GMV / DAU / 转化率 / 留存 / NPS …)
4. **边界在哪**:系统做什么,明确不做什么(non-goals)

**必须输出**:

- 一句话业务定位
- 角色清单与诉求表
- 3–7 条核心业务流程(每条至少给出主路径)
- 5–10 个关键业务指标 + 目标值
- non-goals 清单

**常见误区**:
- ❌ 把"用户登录、下单、支付"当业务流程(这是技术动作不是业务)
- ✅ "买家在限时活动中完成一次秒杀,从看到活动到支付成功 ≤ 5s"

**产出**:`business.md`

## Step 2 — 规模与约束分析

**目标**:把业务需求翻译成**硬约束**,后续方案选型必须被这些约束卡住。

**六类硬约束**:

| 类别 | 要点 |
|------|------|
| **流量** | 峰值 QPS、均值 QPS、读写比、热点比例、突发流量形态 |
| **数据量** | 存量、日增、冷热比、最大单条体积 |
| **延迟** | P50 / P99 / P999 目标;关键链路端到端上限 |
| **一致性** | 强一致(金额、库存)/ 最终一致(Feed、消息)/ 会话一致 |
| **可用性** | SLA(99.9 / 99.95 / 99.99)、RTO、RPO |
| **成本与合规** | 成本上限(机器/带宽/存储)、等保/合规、审计要求 |

**估算方法**:
- 费米估算法:DAU × 人均动作 × 活跃时段比例 = 峰值 QPS
- 2–5 倍安全冗余:峰值 × 2~5 = 设计目标
- 冷热分层:通常 20% 数据贡献 80% 访问

**产出**:`scale-constraints.md`,每类约束必须给出具体数字,不能只说"高并发"。

## Step J — 人类判断文档

**目标**:把 Step 1–2 的业务与约束,转换成人类必须亲自拥有的判断,防止后续 solutions / architecture / harness 只是被动接受 AI 的观点。

**输入**:
- `business.md`:业务定位、角色、流程、指标、边界
- `scale-constraints.md`:流量、延迟、一致性、可用性、成本与合规
- `research.md`:公开证据与争议点

**产出**:`judgment.md`,必须使用 `human-judgment-template.md`。

### 必须回答的 7 类判断

| 类别 | 核心问题 | 示例 |
|------|----------|------|
| 问题定义 | 这个系统到底解决什么,不解决什么? | 秒杀模块是“资格分配系统”,不是“支付系统” |
| 成功语义 | 用户看到的成功,在系统内部到底意味着哪一步完成? | 抢购成功=资格预占成功,不是支付成功 |
| 核心不变量 | 哪些约束任何情况下不能破坏? | confirmed_order <= total_stock |
| 关键取舍 | 我们选择了什么,放弃了什么,为什么愿意承担代价? | Redis Lua 预扣换吞吐,用补偿承担最终一致复杂度 |
| 失败模式 | 哪些失败可接受,哪些必须阻断/补偿/告警? | MQ ACK 失败可重复消费,但不能重复下单 |
| 验收标准 | 什么证据能证明判断成立? | 并发 10w 请求库存 100,成功数恰好 100 |
| AI 委托边界 | 哪些交给 AI,哪些必须人审? | AI 写 Lua,人审不变量和边界测试 |

### 操作流程

1. 先让 AI 基于已有文档预填 `[AI建议]`。
2. 把每条关键判断改写成问题,标为 `[人类判断]`。
3. 在 Gate 1 中要求用户确认、修改或采用默认。
4. 用户确认后,把状态写回 `judgment.md` 和 `assumptions.md`。
5. 后续 `challenges.md`、`solutions.md`、`architecture.md`、`harness/*` 必须引用 `judgment.md` 的关键判断,不能绕过它。

### 反被动接受规则

- 不要只问“是否采用默认”,还要问“你是否能解释为什么”。
- 如果用户选择默认,也要在 `judgment.md` 写明“采用默认的代价”。
- 如果 AI 的建议没有反证条件,必须补上“什么时候这个判断会失效”。
- 如果一个判断不能被测试或压测证明,必须标记为 `[待验证]`。

## ⛳ Gate 1 — 业务边界 + 规模假设 + SLA + 人类判断

**触发时机**:Step J 产出 `judgment.md` 后,进入 Step 3 之前。

**交互模板**:

```
⛳ Gate 1 / 业务边界 + 规模 + SLA + 人类判断

我已经根据【{scenario}】的常识预填了下面的默认假设。以下 N 项是**会影响整个架构走向**且需要人类亲自拥有的判断,需要你拍板:

【默认值清单】
1. MVP 边界:   [默认: 只做 X 和 Y,暂不做 Z]
2. 目标规模:   [默认: DAU=?, 峰值 QPS=?, 存量数据=?]
3. 延迟目标:   [默认: P99 ≤ ? ms]
4. 一致性倾向: [默认: 关键路径强一致,辅路径最终一致]
5. 可用性目标: [默认: 99.95%]
6. 合规红线:   [默认: 无特殊合规 / 等保二级 / 金融级资金审计]
7. 成本上限:   [默认: 未限制]
8. 成功语义:   [默认: 见 judgment.md §1.2]
9. 核心不变量: [默认: 见 judgment.md §2]
10. 不可接受失败: [默认: 见 judgment.md §4]

【回复方式】
- 全部采用默认:回复"采用默认"
- 部分调整:  回复"第 2 条改为 DAU=5000w;第 4 条改为全链路最终一致"等
- 大范围调整:直接贴新的约束清单,我重新对齐
```

**用户回复后**:
1. 更新 `assumptions.md` 与 `judgment.md`,把原来 `[AI建议]` / `[人类判断]` 标签改为 `[已确认:用户]` 或 `[已确认:采用默认]`
2. 简短汇报变更清单
3. 推进 Step 3

## Step 3 — 核心难点提炼

**目标**:从 Gate 1 确定的约束中,逆向推出 3–8 个**决定系统形态**的核心技术难点。

> 📌 **调研驱动**:难点提炼必须参考 `research.md` 中的开源 issue、社区踩坑帖、大厂实战经验。每个难点如有调研佐证,标注 `[调研:来源名]`。

**难点 = 场景 × 约束 组合出的棘手问题**,不是模块。

**10 条难点线索**:

1. **高并发热点**:单点是否会被流量击穿?
2. **库存/资源竞争**:强一致下如何避免超卖 / 重复消费?
3. **一致性边界**:哪些路径需要强一致?哪些可以最终一致?如何补偿?
4. **海量读**:QPS 是否远超 DB 承载?缓存策略?
5. **海量写**:写是否会成瓶颈?分库分表?异步化?
6. **消息可靠性**:消息必达 / 去重 / 顺序 / 堆积?
7. **实时性**:推送链路 / 长连接管理 / 端到端延迟?
8. **存储选型**:结构化 vs 半结构化 vs 时序 vs 图?冷热分层?
9. **可靠性与容灾**:故障隔离、降级、回滚、多活?
10. **安全/防刷/反作弊**:身份、权限、风控、限流?

**难点分类速查表(按业务类型)**:

| 业务类型 | 典型难点 |
|---------|---------|
| 秒杀 | 超卖、热点、限流削峰、防刷、异步落单、库存一致性 |
| IM | 海量长连、消息必达、离线同步、多端一致、群消息扩散、E2E 加密 |
| 支付清结算 | 资金一致、幂等、对账、分布式事务、审计、合规 |
| Feed | 扩散模型(推/拉/推拉结合)、热点、存储、Ranking、冷启动 |
| 推送 | 长连管理、到达率、设备/离线、去重合并、频控 |
| 订单履约 | 状态机、库存预占、逆向流程、超时关单、跨系统一致 |

**每个难点必须给出**:
- 编号 C1..Cn
- 一句话问题定义
- 核心约束(引用 scale-constraints.md)
- 为什么这是决定性难点(不解决系统无法成立)
- 初步方案方向(仅方向,细节放 Step 4)

**产出**:`challenges.md`

## Step 4 — 方案矩阵与决策

**目标**:对每个难点,给出 2–3 个候选方案,做 trade-off,选出推荐方案。

> 📌 **调研驱动**:候选方案必须覆盖 `research.md` 中开源项目已验证的方案。方案矩阵中标注 `[调研:项目名/来源]` 表明该方案有实际验证案例。

**每个难点的方案矩阵结构**:

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 核心思路 | ... | ... | ... |
| 一致性 | 强 / 最终 | ... | ... |
| 吞吐 | ... | ... | ... |
| 复杂度 | 低 / 中 / 高 | ... | ... |
| 运维成本 | ... | ... | ... |
| 失败模式 | ... | ... | ... |
| 适用规模 | ... | ... | ... |
| **推荐** | ✅ / ❌ | ... | ... |
| **理由** | ... | ... | ... |

**trade-off 思维约束**:
- 没有免费的午餐:选了高吞吐必然牺牲点什么(一致性/延迟/复杂度)
- 选型必须回到 Gate 1 的硬约束:违反规模/一致性/成本的方案直接淘汰
- 推荐方案必须标注"它解决了什么,付出了什么代价"

**关键横切方案**(必须单独列一节):
- 流量削峰(排队/令牌桶/漏桶/异步化)
- 缓存策略(多级/旁路/穿透/雪崩/击穿防护)
- 分布式事务(2PC / TCC / Saga / 最终一致补偿)
- 限流熔断(客户端/服务端/网关)
- 可观测性(指标/日志/追踪)

**产出**:`solutions.md`

## Step 4.5 — 难点深度拆解(逐个 C 输出一份 deep-dive)

**目标**:对每一个 `challenges.md` 中的 C{N},单独产出一份 `deep-dives/C{N}-<name>.md` 教学文档。这是把决策**从"扫一眼能看懂"升级到"能上白板讲 40 分钟"** 的关键一步。

> 📌 **职责划分**:
> - `solutions.md` 回答"选哪个",是决策矩阵(架构评审用)
> - `deep-dives/C{N}-*.md` 回答"为什么选它、每一寸是怎么想透的",是教学文档(架构师复盘 / TL 传帮带 / agent 背景知识)
> - 两者**禁止互相复制**

### 为什么必须有这一步

只有 `solutions.md` 的决策矩阵,会导致三类问题:

1. **浅层决策**:矩阵列 A/B/C + 推荐,但"为什么 A 能 work、为什么 B 会崩"没有论证,评审时扛不住追问
2. **agent 上下文贫瘠**:harness prompt 让 agent 去实现推荐方案,但 agent 缺"为什么这么设计"的语境,写出来的代码只学其形不得其神
3. **知识无法沉淀**:决策矩阵像 checklist,读完就忘;教学文档可以反复翻、可以传给新人、可以改编成博客

deep-dives 的存在就是解决这 3 个问题的唯一手段。

### 写作模板(强制 8 节)

每份 `deep-dives/C{N}-<name>.md` 必须严格按以下 8 节结构:

| 节 | 标题 | 核心回答 | 字数参考 |
|----|------|---------|---------|
| 1 | 问题的本质(First Principles) | 为什么这件事根本上难?物理/业务约束是什么? | 400–800 |
| 2 | 关键概念与分类学(Taxonomy) | 讨论这个问题需要哪些词汇?本场景落在哪一侧? | 300–700 |
| 3 | 方案空间的演化(Evolution) | 从最朴素方案开始,每代如何崩溃,下一代如何解决? | 600–1200 |
| 4 | 推荐方案的深度拆解(Deep-Dive) | 为什么 Work + 关键代码 + 工程陷阱 + 边界与二阶问题 | 1000–2000 |
| 5 | 与其他难点的交互(Cross-Challenge) | 这个方案如何传染到 C{x}?C{x} 如何反过来约束这里? | 300–600 |
| 6 | 业界真实案例(Industrial Evidence) | 至少 2 个不同规模的真实案例,引用 research.md | 400–800 |
| 7 | 常见反模式与误区(Anti-Patterns) | 看似合理但会崩的方案 × 3–5 个 | 400–800 |
| 8 | 工程落地检查清单(Checklist) | 设计 → 实现 → 测试 → 上线的可勾选 checklist | 200–500 |

完整模板、每节的写作要求、反模式示例、自检清单见 `deep-dive-template.md`。

### 执行流程

```
1. 从 challenges.md 拉取所有 C{N}
2. 对每个 C{N} 并行/顺序生成 deep-dives/C{N}-<name>.md
   ├─ 引用 challenges.md §C{N} 作为问题输入
   ├─ 引用 solutions.md §C{N} 的推荐方案作为深拆对象
   ├─ 引用 research.md 中相关来源作为案例依据
   └─ 严格按 8 节模板产出
3. 每份产出后过自检清单(见 deep-dive-template.md 末尾)
4. 全部产出后,统一做一次"跨难点一致性扫描":
   └─ 检查每份 deep-dive 的第 5 节(跨难点交互)是否双向闭合
      (C1 说影响 C3 → C3 的文件里第 5 节也应提到与 C1 的关系)
```

### 关键写作规则(硬约束)

| 规则 | 要求 |
|------|------|
| **不是方案矩阵** | 不允许再列 A/B/C,deep-dive 聚焦**推荐路径**的深拆 |
| **不是 solutions.md 复印** | 不允许重复 solutions.md 的 trade-off 表 |
| **必须有追问链条** | 第 1 节的 first-principles 论证必须用"为什么 → 因为 → 那为什么 → 因为..."的追问结构 |
| **必须有演化叙事** | 第 3 节至少 3 个阶段,每阶段带规模数字 + 崩溃机制 |
| **必须有实战细节** | 第 4 节至少 30 行代码/SQL/Lua + 3 个"症状→原因→修复"格式的陷阱 |
| **必须有数字** | 每节至少 2 个具体数字(QPS/ms/字节/百分比) |
| **必须有引用** | ≥ 3 处 `[R-XX]` 引用 research.md;≥ 1 次 challenges.md / solutions.md 引用 |
| **必须有图** | ≥ 1 张 Mermaid(时序/架构/状态机/流程) |

### 反模式清单(写出下列内容即判不及格,回炉重写)

- ❌ 只有标题列表,每节只有 2–3 句话
- ❌ 把 solutions.md 的 trade-off 表直接粘过来
- ❌ 8 节之一被跳过或合并
- ❌ 用"合理/高效/可靠/稳定"替代具体论证
- ❌ 案例只提"某大厂"但没有 research.md 引用、没有规模数字
- ❌ 单份 < 2500 字
- ❌ 8 节中包含"方案 A / 方案 B / 方案 C"的决策矩阵样式

### 进度与工作量估算

| 难点数量 | 预计单次产出 deep-dives 总字数 | 适合做法 |
|---------|-----------------------------|---------|
| 3 个 | 8k–15k 字 | 一次性全部产出 |
| 4–6 个 | 15k–30k 字 | 按 C{N} 顺序,每产出一份就做自检,确保质量一致 |
| 7–8 个 | 25k–45k 字 | 分两批:先核心 3–4 个,过一轮质量再出剩余 |

### 产出

`deep-dives/C1-<name>.md`,`deep-dives/C2-<name>.md`,... — 每个 challenge 一份,命名用 kebab-case 简短描述(如 `C1-stock-deduction.md`)

### 验收标准

- **数量**:与 `challenges.md` 中 C{N} 数量 1:1 对应,不缺不多
- **结构**:每份严格 8 节
- **深度证据**:过 `deep-dive-template.md` 末尾的 16 条自检清单
- **金线**:读完任一份 deep-dive,中级工程师能在白板做 40 分钟脱稿分享

---

## ⛳ Gate 2 — 一致性策略 + 容灾等级 + 技术栈

**触发时机**:Step 4.5 所有 deep-dives 产出后,进入 Step 5 之前。

> 📌 **为什么放在 4.5 之后**:deep-dives 的第 3 节(方案演化)和第 4 节(边界与二阶问题)会揭示"某个方案在当前规模下其实撑不住""这个技术栈的选择要匹配某个一致性模型"等关键证据。在没有 deep-dives 的情况下做 Gate 2 决策,用户缺证据;有了 deep-dives,Gate 2 的选项更具象,也更容易拍板。

**交互模板**:

```
⛳ Gate 2 / 一致性 + 容灾 + 技术栈

基于 N 个核心难点的方案对比 + 每个难点的深度拆解,以下是影响后续架构蓝图的关键选择,需要你拍板:

【一致性策略(逐路径)】
- 路径 1 [X]:[默认: 强一致,用 DB 事务]
- 路径 2 [Y]:[默认: 最终一致,消息 + 补偿]
- ...

【容灾等级】
- 默认: 同城双活 / 主备 / 单机 + 冷备
- 请确认,或给出具体等级

【技术栈(影响 harness prompt)】
- 语言:       [默认: Go / Java / TypeScript]
- 关系库:      [默认: PostgreSQL / MySQL]
- KV 缓存:     [默认: Redis]
- 消息队列:    [默认: Kafka / RocketMQ]
- 搜索(如需): [默认: Elasticsearch]
- 部署:        [默认: Kubernetes + 云厂商托管]

【回复方式】
- 采用默认 / 逐项调整 / 贴新栈
```

**用户回复后**更新 `assumptions.md`,推进 Step 5。

---

## Step 5 — 架构蓝图

**目标**:把前 4 步的结论落成可视化架构,agent 后续读这份架构来写代码。

**必须包含**:

1. **系统架构图**(Mermaid `graph TB`):前端/网关/业务层/存储/基础设施分层
2. **核心流程时序图**(Mermaid `sequenceDiagram`):至少 2 张(最复杂的 2 条主路径)
3. **模块分解表**:模块编号 M01..Mn、职责、依赖、技术栈、对应难点
4. **数据模型**:核心实体表(`classDiagram` 或表格 + 约束)
5. **关键接口契约**:对外 API + 模块间接口的签名、错误码、幂等性
6. **技术栈应用图**:哪些组件用哪些技术

**模块划分原则**:
- 单一职责,接口稳定
- 边界与 `challenges.md` 中的难点对齐(通常每个难点落在 1–2 个模块)
- 模块数量 5–15 个最佳,过多切得太碎,过少耦合过重

**产出**:`architecture.md`

---

## Step 6 — 实施路径

**目标**:把架构切成**可并行 / 可验收**的阶段,并为 harness 粒度决策做铺垫。

**必须包含**:

1. **MVP 切片**:最小可运行子集(通常 2–4 个核心模块)
2. **迭代阶段**:Phase 1 / 2 / 3 每阶段包含哪些模块、什么验收标准
3. **里程碑时间线**(Mermaid `gantt` 可选)
4. **验收标准**:功能验收 + 性能验收(对齐 Gate 1 SLA)
5. **风险矩阵**:至少 5 条,含触发条件 + 影响 + 应对

**产出**:`playbook.md`

---

## ⛳ Gate 3 — MVP 范围 + harness 粒度

**触发时机**:Step 6 产出后,进入 Step 7 之前。这是最关键的一个 Gate,直接决定后续 harness 规模。

**交互模板**:

```
⛳ Gate 3 / MVP 范围 + harness 粒度

架构已有 N 个模块。为了生成合适规模的 harness prompt 包,需要你拍板:

【MVP 范围】
默认 MVP 包含模块: [M01, M02, M03]
其余模块 [M04..Mn] 作为后续迭代

【Harness 粒度】
- 目标 coding agent:   [默认: {target_agent}]
- 模块 prompt 数量:    [默认: 等于 MVP 模块数 = N]
- 每模块任务拆分粒度:   [默认: 接口级,每模块 3–6 个 task prompt]
  ├─ 粗粒度(2–3 task/模块):快,但 agent 需要强推理能力
  ├─ 中粒度(3–6 task/模块):推荐默认
  └─ 细粒度(6–12 task/模块):最稳,但 prompt 多
- 是否生成测试 prompt:  [默认: 每个 task 内置 DoD,不另起 test 模块]
- 是否生成部署 prompt:  [默认: 放入 master-prompt 附录,不单独建模块]

【回复方式】
- 采用默认
- 调整 MVP 包含的模块
- 调整粒度:"按细粒度拆分"
```

**用户回复后**更新 `assumptions.md`,推进 Step 7。

---

## Step 7 — Harness 产出

**目标**:把前 6 步所有决策压进一套分层 prompt 包,放到 `harness/` 下。

**产出顺序**:

1. `harness/master-prompt.md`:
   - 项目背景(引用 `business.md`)
   - 全局约束(引用 `scale-constraints.md` + `assumptions.md`)
   - **参考实现**(引用 `research.md` 中的开源项目,作为 agent 的架构参考)
   - 整体架构图(从 `architecture.md` 复制或引用)
   - 模块清单与执行顺序
   - 全局代码/目录规约
   - 质量门(测试/压测/文档)

2. `harness/modules/M{N}-<name>.md`:
   - 对每个 MVP 内模块,产出一份
   - 内容:职责、依赖、对应难点、内部组件、输入/输出契约、验收、任务索引

3. `harness/tasks/M{N}-T{M}-<name>.md`:
   - 对每个模块的任务,按 Gate 3 粒度产出
   - 内容:任务目标、前置条件、输入/输出契约、实现要点、边界/异常、DoD、最小测试

4. 回写 `master-prompt.md` 的模块与任务索引

**详细模板见** `harness-prompt-templates.md`,**设计哲学见** `harness-prompt-design.md`。

**自检清单**:见 `SKILL.md` 的"质量闸门"章节。

**交付格式**:

```

📚 深度拆解(N 份,教学型)

  - 执行过程中若发现新约束,回来更新 assumptions + 相关模块 prompt
```

## 继续指令映射

| 用户指令 | skill 行为 |
|---------|-----------|
| "继续" | 进入下一 Step,或在 Gate 时提示先确认 |
| "跳过 Gate,全部默认" | 按默认值更新 assumptions,直接推进 |
| "在 solutions 加一个方案 C" | 只重算 solutions.md 对应小节,不动其他 |

| "task 粒度改细" | 重拆 Gate 3 粒度决策,重算 harness/tasks/ |
| "切换 agent 到 codex" | 更新 assumptions,按 codex 特性调整 master-prompt 头部 |
