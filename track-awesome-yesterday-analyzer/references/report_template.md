# Report Template

Use this as the default structure for the final answer.

# Track Awesome List 昨日更新深度分析报告（YYYY-MM-DD）

## 1. 昨日更新速览
用 2 到 4 段完成开场：
- 说明昨天整体更新规模与更新面向
- 点出最值得关注的主题信号
- 说明这份报告先做全量概览，再做重点深挖

## 2. 昨日更新全量概览
这一节必须覆盖“昨天实际更新了什么”，而不是精选。

建议按下面的颗粒度组织：
- 先按 awesome list 列出
- 每个 list 下说明当天新增了哪些条目
- 如果单个 list 新增很多，可以合并成“新增方向 + 代表条目”写法，但不能跳过

推荐小标题格式：
### 2.x <awesome list 名称>
先用一小段说明这个 list 昨天更新了什么，再列出新增条目或新增方向。

## 3. 值得重点解读的更新
从昨天的新增里选择少量最值得分析的内容，通常 3 到 6 组即可。

每组建议回答这些问题：
- 它是什么
- 它解决什么问题
- 为什么它在昨天这批更新里尤其值得看
- 它对工程、产品、基础设施或生态意味着什么

避免把这一节写成项目说明书。重点是“为什么值得深入解读”。

## 4. 趋势信号与生态解读
把前面分散的更新串起来，回答：
- 昨天这些更新共同说明了什么变化
- 哪些方向正在加速
- 哪些只是热闹、哪些可能形成持续趋势
- 对不同角色各意味着什么，例如产品工程师、AI 应用开发者、基础设施团队、独立开发者

这里要有观点，但要以第二节和第三节的事实为基础。

## 5. 后续建议与跟踪清单
给出可执行建议，而不是空泛结论。

建议至少包含：
- 接下来值得继续跟踪的 3 到 5 条主题线
- 哪些项目适合亲自试用
- 哪些方向值得纳入团队观察清单
- 如有必要，可补一段“本次分析的边界与不确定性”

## Writing guidance

- 多用自然段，少用纯罗列。
- 开场简洁，正文展开。
- 第二节重事实覆盖，第三和第四节重判断。
- 如果发现昨日更新量特别大，可以在第二节先分主题再分 list，但仍要保证没有把更新内容偷换成精选。
